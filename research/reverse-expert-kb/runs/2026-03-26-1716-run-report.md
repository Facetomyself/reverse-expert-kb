## New findings
The main practical refinement preserved this run is:
- Apple-platform XPC cases should not flatten everything into one vague story of “the XPC call happened” or “the service recovered”

The branch should preserve four separate proof objects:
1. **accepted**
2. **replied**
3. **reconnected**
4. **consumed**

That matters because two different practical subcases recur:

### Subcase A: reply/error truth is already the strongest truthful boundary available
If the service seam is mostly useful as contract/lifecycle evidence, then:
- successful reply, explicit error, interruption, or invalidation may be the strongest truthful object for the current step
- the analyst should preserve it as reply/error truth without overclaiming that the same durable service-owned semantic consequence has already been proved

### Subcase B: restart/reconnection creates fake progress
If `launchd` restarts the service or the connection becomes valid again, then:
- reconnection truth is still weaker than proving that the same exported-object method or later service-owned reducer actually ran again
- the practical stop rule is to freeze one exported-object method re-entry or one later durable effect before claiming the behavior repeated
- do not stop at generic “the connection came back” evidence

A second useful refinement preserved this run:
- listener/service lookup and acceptance are still routing truth even when they are stable across runs
- lifecycle recovery does not automatically collapse into consumer truth

## Source-backed synthesis
The useful cross-source synthesis here is:
- Apple’s archived XPC-services guide makes the lifecycle model explicit: XPC services are launchd-managed, can be restarted after crash, and can be terminated when idle
- that same source also reinforces that `NSXPCConnection` is an RPC-style proxy/exported-object architecture, which naturally separates route setup from remote semantic work
- the Apple documentation/result surfaces for `NSXPCConnection`, `exportedObject`, `interruptionHandler`, and listener acceptance help preserve client-side route truth, interruption/invalidation truth, and exported-object consumer truth as distinct proof objects

The resulting branch memory is sharper:
- not just “XPC exists”
- but “acceptance, reply/error, reconnection, and durable service-owned consequence are distinct proof layers; stop at the smallest one that actually predicts behavior”
