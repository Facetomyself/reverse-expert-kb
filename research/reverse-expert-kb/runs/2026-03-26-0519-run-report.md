## New findings
### 1. Windows RPC replay fixtures can still be dishonest after interface/opnum/argument recovery
The source set reinforced a narrow but important workflow claim:
- binding handles are part of the replay object, not just setup trivia
- context handles are not interchangeable ambient tokens
- a visible opnum plus plausible arguments is still not a truthful fixture if binding family, object UUID, authn posture, or context-handle lineage differ from the live call

### 2. `NdrClientCall*`-adjacent boundaries are often the cheapest honest constructor path
The Microsoft RPC stub material plus field writeups support a practical boundary choice:
- prefer one generated/helper/stub invocation path that still preserves opnum and argument shape
- do not jump to whole-transport recreation if the real first honest constructor already exists near `NdrClientCall*`

### 3. Some apparent “argument uncertainty” is really call-comparability uncertainty
This run sharpened the split between:
- method-contract truth
- call-context truth
- later acceptance-gate truth

Practical consequence:
- if a visible context handle already exists, replay failure may be better explained by invalid handle lineage/interface ownership than by wrong scalar/string arguments
