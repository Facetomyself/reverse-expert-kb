## New findings
The main practical finding from this run is that the iOS Swift-concurrency branch needed a sharper canonical reminder that **resume truth, resumed-consumer truth, and MainActor-side consumer truth are adjacent but not identical proof objects**.

More concretely:

1. **Imported `async` visibility is weaker than consequence ownership**
   - bridged async entry points may still just wrap an older completion-handler family
   - therefore “async function found” is not yet a strong stopping boundary

2. **Exact-once continuation discipline creates analyst-meaningful failure shapes**
   - callback visibility
   - continuation creation/storage
   - actual resume
   - missing-resume leak/suspend
   - double-resume misuse
   - cancellation-owned conclusion
   should not be collapsed into one vague “async drift” story

3. **Resume truth is still weaker than resumed reducer truth**
   - a continuation can resume without yet proving which resumed reducer, mapper, or task-side consumer changes behavior

4. **MainActor/UI-state truth can still be one hop later than resumed reducer truth**
   - when the case is strongly UI/view-model/coordinator shaped, the first practical consequence boundary may be one MainActor-isolated state write or route handoff rather than the raw resume site itself

5. **Annotation visibility is weaker than consumer proof**
   - seeing `@MainActor` is useful, but should not by itself be overread as the first real behavior-changing consumer without one actual state write / route selection / later effect

## Practical synthesis added to the KB
The practical refinement preserved this run is:

```text
callback/delegate truth already good enough
  -> ask whether imported async is only a bridge face
  -> separate continuation creation/storage from actual resume
  -> separate actual resume from first resumed task-side reducer
  -> when UI/coordinator-owned, separate resumed reducer from first MainActor-side state consumer
  -> stop at the first consumer that predicts one later effect
```

And the compare-pair reminder preserved this run is:

```text
same callback family visible
  != same continuation created/stored correctly
  != same continuation resumed exactly once
  != same resumed reducer reached
  != same MainActor-side state consumer fired
```

That improves operator value because it gives the iOS branch a more truthful stop rule for modern Swift-heavy cases instead of letting them blur into generic callback hunting or generic async language.
