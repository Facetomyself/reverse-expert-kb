## 2. New findings

### Symbol/type/signature recovery deserves a dedicated topic page, not just a subsection under decompilation
The strongest result from this run is structural.

Earlier passes already hinted that decompilation evaluation and symbol/type recovery are not the same thing. This run added enough direct evidence to make that separation much firmer.

A useful split now looks like:
- **decompilation quality** — semantic correctness, readability, recompilability
- **type recovery** — trustworthy types, struct layouts, prototypes
- **symbol recovery** — names and semantic anchors
- **signature recovery** — parameters/returns/call-shape reconstruction

That separation is much closer to expert analyst reality than a flat “better decompilation” bucket.

### R3-Bench is the strongest current anchor for symbol recovery as its own benchmark family
The best new source this run was the ASE 2025 abstract page for:
- **R3-Bench: Reproducible Real-world Reverse Engineering Dataset for Symbol Recovery**

Important extracted details:
- introduces **AST-Align** for aligning variables and struct-access expressions between source and binary
- spans **x86 and ARM** and **C/C++/Rust**
- claims **4× more struct fields** than prior alignment methods
- frames the dataset as **metadata-rich**, **extensible**, and **reproducible**
- reports **over 10 million functions** across multiple architectures
- baseline evaluation ranges from **n-gram models to LLMs**
- notable result: general LLMs initially perform poorly but improve sharply with proper demonstration

This is important because it directly ties symbol recovery to:
- alignment quality
- dataset reproducibility
- semantic-vs-syntactic recovery distinction
- prompt/demo sensitivity for LLM-based methods

### Type inference is becoming benchmarked directly, not just argued about informally
A second strong new signal:
- `sefcom/decompiler-types-benchmark`
- the associated SURE 2025 paper **Benchmarking Binary Type Inference Techniques in Decompilers**

What is high-confidence from this run:
- the benchmark repository is public
- it explicitly contains binaries, extraction scripts, evaluation scripts, extracted type data, and results
- search-layer metadata associates it with **Hex-Rays, Binary Ninja, Ghidra, angr, and Retypd**

Even though detailed metric extraction from the PDF remains incomplete, the public benchmark repo itself is already important. It suggests this area has crossed from “tool opinion” into reproducible comparative evaluation.

### Practical analyst value of type recovery is clearer when seen through BTIGhidra
Trail of Bits’ BTIGhidra write-up was especially useful as a bridge between benchmark abstractions and analyst experience.

It highlights that stronger type inference:
- turns opaque pointer arithmetic into field access and array indexing
- reduces `void*` clutter
- propagates user-provided type information inter-procedurally
- synthesizes composite/recursive types on stripped binaries

The deeper synthesis is that type recovery quality changes the analyst’s **navigation surface**, not just cosmetic pseudocode quality.

### XTRIDE suggests type/signature recovery is splitting into accuracy vs deployability questions
Another valuable new source was:
- **Practical Type Inference: High-Throughput Recovery of Real-World Structures and Function Signatures** (XTRIDE, 2026)

Important extracted claims:
- **90.15%** overall type inference accuracy on DIRT
- **2300×–7070×** faster than some heavier prior approaches
- adds confidence scores for threshold-based filtering
- extends the approach to **function signature recovery** in embedded firmware

This matters because expert workflows often need selective automation. A slower, aggressive system that pollutes the type database may be worse than a fast system that abstains when uncertain.

### LLM symbol recovery may be unusually sensitive to demonstrations and context
One subtle but important finding from the R3-Bench abstract is that LLM effectiveness improved dramatically with proper demonstrations.

That suggests an expert-KB hypothesis worth tracking:
- symbol recovery may depend heavily on exemplars, local project conventions, and already-recovered context
- benchmarking raw model performance without demonstration/context may understate real analyst-in-the-loop potential
- but it also means naive one-shot LLM comparisons may be misleading

## 4. Reflections / synthesis
The KB structure is getting sharper.

This run strengthened the case that reverse-engineering evaluation should not be represented by a single “binary understanding” bucket. At minimum, the KB now has strong support for separating:
- decompilation output quality
- symbol/type/signature recovery
- task-level binary understanding
- workflow-fit / analyst-support patterns

A second synthesis is that semantic anchors matter disproportionately in expert RE. Analysts can often work around ugly pseudocode, but missing or misleading names/types/prototypes poison many later steps at once.

A third synthesis is that practical deployment properties matter here more than they first appear. Throughput, confidence calibration, and conservative abstention are not engineering footnotes; they are central to whether recovery systems help or hinder expert workflows.
