# Reverse Expert KB Run Report — 2026-03-14 09:00 Asia/Shanghai

## 1. Scope this run
This run focused on a structural gap in the KB’s workflow coverage: **analytic provenance, evidence-trail support, and cognitive-process capture** for reverse engineering.

I explicitly avoided another benchmark-heavy pass and instead targeted material that helps answer:
- how expert reverse engineers preserve context over long investigations,
- how RE activity can be captured and studied empirically,
- how evidence externalization should fit the KB’s workflow spine,
- whether this seam justifies a future standalone topic page.

## 2. New findings

### A. Analytic provenance is a real RE-specific support category, not just a generic note-taking idea
The strongest new signal this run is that reverse engineering has direct literature on **analytic provenance** rather than only adjacent HCI/sensemaking concepts.

From the SensorRE / AFIT dissertation thread:
- reverse engineering is explicitly framed as an open-ended exploration problem with multiple competing explanations,
- one of the core burdens is preserving the context of findings within the larger task,
- provenance capture plus replay/visualization (graph/storyboard style views) is treated as support for analyst exploration rather than mere auditing.

This matters because it strengthens a KB claim that had previously been more inferential than direct: **expert RE depends on evidence externalization and history preservation as part of the core workflow**.

### B. reAnalyst turns RE workflow observation into a scalable methodological object
The reAnalyst paper adds a newer layer beyond one-off workflow studies:
- tool-agnostic collection of screenshots, keystrokes, active processes/windows, and optional annotations,
- semi-automated extraction of higher-level RE activities from lower-level traces,
- explicit concern with scaling RE-practice studies beyond manually coded small datasets.

This is useful for the KB in two ways:
1. it is new evidence that workflow study itself is becoming more instrumented and reusable;
2. it suggests the KB should track not only workflow findings, but also the **pipelines used to study workflow**.

### C. The workflow topic should explicitly distinguish notebook/memory support from provenance support
Previously, the workflow page already covered notebook/memory-augmented analysis in broad terms.
This run suggests a sharper split:
- **notebook / memory support** = analyst-authored externalization,
- **analytic provenance** = system-captured history of actions, transitions, artifacts, and evidence trails.

These overlap, but they are not identical. The second deserves explicit representation in the ontology and future topic planning.

### D. There is now a stronger cross-domain bridge between software RE cognition and hardware RE cognition
The hardware RE mixed-methods paper (`I see an IC`) suggests that cognitive/process capture is not only a software-binary concern.
That means this seam may eventually support a broader synthesis:
- software RE process models,
- hardware RE problem-solving observation,
- provenance/evidence-trail systems,
- immersive/representation-aware support.

That is a richer and more cumulative direction than simply adding more LLM-sidekick anecdotes.

## 4. Reflections / synthesis
This run improved the KB more by **tightening ontology** than by broadening coverage.

The main synthesis change is:
> the workflow spine should explicitly include **analytic provenance / evidence-trail support** as a first-class analyst-support pattern.

That gives the workflow topic a more convincing internal decomposition:
- process models of RE,
- notebook/memory support,
- provenance/evidence-trail capture,
- tooling/interface mediation,
- human–LLM collaboration,
- workflow evaluation.

This also improves the overall KB by making an important distinction:
- some workflow support is about helping the analyst think and write,
- some workflow support is about **capturing and reconstructing the path of analysis itself**.

That second category has clear downstream relevance for:
- long-horizon investigations,
- handoff/review,
- empirical study of expert behavior,
- trust calibration and correction of earlier hypotheses.

In other words, this run strengthens the claim that reverse-engineering expertise is not just recovering the next object; it is also preserving enough of the reasoning path that future steps remain grounded.
