## New findings
The practical refinement added this run is:
- wrapper-heavy and durable-transfer comms cases need one sharper split than the malware branch had been preserving canonically
- visible initiation is often weaker than true commitment
- durable transfer progression is often weaker than final durable availability/effect

A compact operator chain preserved this run is:

```text
request family plausible
  -> initiating wrapper call is visible
  -> callback/state progression proves whether the wrapper really advanced
  -> durable queue/job ownership proves whether later transfer behavior belongs to this request
  -> final completion or file/effect availability proves the consequence boundary
```

More concrete practical refinements:
- for WinHTTP-style async wrappers, `SendRequest(...)` alone is weaker than callback-confirmed request-context progression
- for URLMon-style helpers such as `URLDownloadToFile(...)`, a successful start is weaker than callback-confirmed progress / data / stop-binding truth
- for BITS-style abuse, preserve separate truth for:
  - job creation
  - job resume/ownership
  - callback or transferred-state truth
  - `Complete()` / final renamed-file availability

This gives the malware branch a better stop rule:
- do not overread initiation as commitment
- do not overread queued or in-progress durable transfer state as the same proof object as durable file/effect consequence

## Reflections / synthesis

The useful cross-source synthesis is:
- wrapper APIs often expose a compressed state machine rather than one transparent send primitive
- callback stages are often the truthful proof object for async APIs
- durable transfer APIs separate transfer ownership/progression from final usable consequence
- therefore one honest malware-side handoff often lives at a smaller state or completion boundary than an analyst first expects

This is good branch memory because it tells the operator where to stop and where **not** to stop.

## Concrete scenario notes or actionable tactics added this run
Preserved practical tactics:
- if the API family is async wrapper-shaped, prefer a callback-confirmed request-context transition over the initiating helper call
- if the API family is durable-transfer-shaped, write the case notes in four separate truth buckets:
  1. create
  2. resume/ownership
  3. callback/transferred state
  4. `Complete()` / final availability
- do not hand off into broad protocol narration until one of those smaller commitment boundaries is frozen
- when a file-bearing consequence matters, prefer the first durable availability boundary over generic “download started” language
- when a compare pair exists, test whether the same request/job owner reaches the same progression/completion state rather than only whether the same helper was called
