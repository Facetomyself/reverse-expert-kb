## New findings
The main practical refinement added this run is:
- a localized first bad write or decisive reducer is **not always** the right stopping point
- but the fix is also **not** to keep browsing trace history indefinitely
- the correct continuation is often **zero or one more hop**:
  - stop immediately if the write/reducer already makes the next branch-specific task obvious
  - otherwise carry the case exactly one more hop to the first downstream consumer/consequence edge that operationalizes that state
  - then hand off to the narrower branch-specific proof question

A compact operator chain preserved this run is:

```text
visible bad late object
  -> narrow watched object
  -> first useful write or decisive reducer
  -> if that already names the next branch-specific task, hand off
  -> otherwise carry the case one more hop to the first downstream consumer/consequence edge
  -> then hand off
```

Another useful refinement preserved this run:
- in TTD/query-heavy workflows, watched-object choice and query scope are part of **truth selection**, not only performance tuning
- an over-broad watched object can return technically real memory events that still fail to answer the operator’s actual question

## Concrete scenario notes or actionable tactics added this run
Preserved practical tactics:
- if a localized policy bucket write already cleanly partitions good/bad or accepted/rejected outcomes, stop there and route to the relevant branch-specific note
- if the write is still too local to explain behavior, ask: **who operationalizes this state first?**
- only carry the case one more hop to that first operationalizing consumer/consequence edge
- in TTD/query-heavy workflows, shrink the watched object before widening time range or memory range
- treat oversized TTD memory/call queries as a workflow smell, not just a performance mistake
