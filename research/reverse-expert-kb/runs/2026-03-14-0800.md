# Reverse Expert KB Run Report — 2026-03-14 08:00 Asia/Shanghai

## 1. Scope this run
This run focused on a genuinely new axis that was underrepresented in the KB so far:

- **mobile reverse engineering as a distinct expert domain**
- **runtime instrumentation as workflow infrastructure**
- **Android/iOS layer selection for analysis**
- **trace-driven inquiry with Frida/Stalker-style tooling**
- **environment/access constraints** (root, jailbreak, gadget, virtualization)
- **iOS arm64e / PAC-era mitigation-aware analysis** as an emerging expert concern

The aim was not to collect generic mobile security material, but to extract what is structurally different about expert mobile reversing compared with desktop/native reversing.

## 2. New findings

### Mobile reverse engineering should be treated as a first-class domain, not a platform variant
The strongest synthesis from this run is that mobile reversing is not just ordinary binary analysis applied to APKs or IPAs.

It is shaped by:
- mixed runtime layers
- platform access constraints
- heavy dependence on runtime instrumentation
- anti-debugging / anti-instrumentation friction
- environment-control questions that materially affect what analysis is even possible

That means the KB should not leave “mobile” as a single line item under domains. It deserves explicit structure.

### Runtime instrumentation is a core analysis primitive in mobile RE
The OWASP MASTG Frida material was valuable because it states the tooling model plainly.

Frida exposes several distinct capabilities that map directly onto analyst workflows:
- **Interceptor** for targeted hooks
- **Stalker** for high-granularity code tracing
- **Java** bridge for Android runtime objects/classes
- **ObjC** bridge for iOS runtime objects/classes

This suggests a better model for mobile reversing is:
- static analysis
- runtime foothold acquisition
- runtime instrumentation
- iterative layer-specific questioning

That is closer to lived practice than a static-only framing.

### Instrumentation mode choice is itself an expert decision
The combination of the OWASP Frida page and the Frida Stalker docs adds a useful distinction.

`Interceptor` and `Stalker` are not interchangeable conveniences:
- Interceptor is direct and practical for targeted hooks, but can be more detectable and may overwrite code in ways relevant to anti-debug checks.
- Stalker uses dynamic recompilation/copying of code blocks, supports high granularity, and is explicitly positioned around transparency/performance tradeoffs.

That means mobile RE expertise includes choosing among:
- targeted hooks
- function tracing
- block/instruction tracing
- stealth vs convenience
- overhead vs observability

### Mobile targets force an early decision about which layer to interrogate
A useful structural insight from this run is that mobile analysis often spans three layers:
- **managed / framework layer**
  - Java/Kotlin on Android
  - Objective-C/Swift on iOS
- **native layer**
  - JNI / NDK libraries / C/C++ components
- **platform mediation layer**
  - sandboxing, signing, loaders, entitlements, private APIs, anti-debugging, IPC, dyld behaviors

This feels important because many practical analyst questions are answered fastest at the framework/runtime layer, while others only resolve in native code or in platform behavior.

So “which layer do I instrument first?” is a meaningful expert choice.

### iOS reversing is constrained as much by environment as by code complexity
The Corellium article is vendor-oriented, but still useful as a structural source.

It reinforces that iOS RE is heavily shaped by:
- code signing
- hardware-backed protections
- jailbreak restrictions
- anti-debugging and tooling detection
- the growing importance of virtualization or otherwise controlled environments

That supports a KB framing where **environment control** is part of reverse-engineering expertise, especially on iOS.

### Modern iOS reversing increasingly overlaps with mitigation-aware analysis
The Epsilon Security write-up on CVE-2025-31201 was the most valuable “new-seam” source in this run.

Even from partial extraction, it clearly shows that modern iOS reversing increasingly involves:
- Mach-O and loader reasoning
- dyld interposing behavior
- Objective-C method/IMP handling
- authenticated pointers and diversifiers
- subtle page-permission and PAC-related behavior

This suggests a new subtopic for the KB:
- **arm64e / PAC-aware reversing and mitigation-aware patch diffing**

That is a distinct expert concern and should not be folded invisibly into generic iOS notes.

### “Hookability” is a separate analyst-relevant property from “readability”
One of the best synthesis points from this run is that mobile targets need to be judged along at least two dimensions:

- **understandability**
  - how readable static output is
  - how recoverable logic is from decompilation/disassembly
- **hookability / traceability**
  - how easily the analyst can place, sustain, and trust runtime observations
  - whether the target detects or destabilizes instrumentation
  - whether the right runtime layer is accessible

That second dimension seems especially important in mobile RE.

### Toolchain drift is part of practical mobile expertise
The Frida 17 notes are a reminder that mobile reversing is unusually sensitive to tooling drift.

Bridges, APIs, packaging, and assumptions in public scripts can change quickly. That means expert knowledge includes:
- keeping scripts/tooling current
- distinguishing stale tutorials from current workflows
- maintaining instrumentation infrastructure as part of the craft itself

## 4. Reflections / synthesis
This run improved the KB in an important way: it added a domain where **analysis-environment constraints** are not side issues but part of the core intellectual problem.

Earlier runs already established strong topic families around:
- decompilation evaluation
- symbol/type/signature recovery
- obfuscation/deobfuscation and packed binaries
- firmware/protocol context recovery
- analyst workflows and human–LLM teaming

This run suggests another first-class family:
- **mobile reversing and runtime instrumentation**

A useful decomposition now looks like:
- Android vs iOS runtime models
- dynamic instrumentation patterns
- access strategy (root/jailbreak/gadget/virtualization)
- anti-instrumentation and anti-debugging friction
- mitigation-aware analysis (especially modern iOS PAC-era behavior)

Another useful synthesis is that some reverse-engineering domains are best understood through **runtime answerability**, not just static recoverability. Mobile reversing appears to be one of them.
