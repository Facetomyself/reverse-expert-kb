## New findings
Retained practical branch rule:

```text
registered
  != ready / woken
  != delivered / re-armed truth
  != first loop-owned consumer proved
  != later visible consequence truth
```

Concrete practical reinforcements added this run:
- `epoll_ctl(...)` registration is weaker than relevant readiness truth
- one `epoll_wait(...)` return is weaker than delivery/re-arm truth and weaker than first loop-owned consumer proof
- visible `eventfd` wakeup is weaker than the first pending-work reducer or loop callback that actually makes the wakeup behaviorally meaningful
- loop-framework identity (`epoll`, libuv, libevent, libev) should not silently replace first callback/dispatcher ownership proof
- edge-triggered / one-shot cases need explicit preservation of drain/re-arm truth instead of collapsing one returned event into one solved consumer story
