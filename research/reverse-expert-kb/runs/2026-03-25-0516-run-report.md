## New findings
The main practical finding from this run is that the runtime-evidence branch needed a sharper explicit distinction between two adjacent proof objects:

1. **First meaningful divergence**
   - where the good/bad runs first separate in a behavior-bearing way

2. **Best watched object**
   - the smallest durable field / slot / slice / reducer output that reverse watchpoint, TTD memory-query, or bounded backward-search workflows can answer efficiently

The KB already handled noisy early mismatches well.
What it still benefited from was preserving the reminder that these two objects are often adjacent but not identical.

Concrete operator traps this run tried to prevent:
- treating a good divergence narrative as already watchpoint-ready
- staying in compare-run improvement mode after the pair is already trustworthy enough
- watching an oversized callback object, request object, or policy/result family when only one reduced slot actually predicts the consequence
- widening memory/query scope before shrinking the target object

## Practical synthesis added to the KB
The key bridge now preserved more explicitly is:

```text
compare pair becomes trustworthy
  -> first meaningful divergence becomes visible
  -> ask whether that divergence is already a good watched object
  -> if not, shrink it into one field / slot / slice / reducer output
  -> use reverse watchpoint / memory-query / bounded backward search there
  -> localize one first useful write or decisive reducer
```

That improves operator value because it changes what the analyst does next:
- stop diffing sooner when the pair is already good enough
- ask for the smallest durable object the debugger can answer well
- let watched-object reduction, not endless compare refinement, carry the next step
