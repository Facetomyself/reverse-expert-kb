## Practical operator value added
This run strengthened a live-case rule that is easy to get wrong:
- a visible Apple XPC route is not the same thing as a visible reply
- a visible reply/error is not the same thing as a later healthy-looking reconnection
- a later healthy-looking reconnection is not the same thing as one service-owned consumer actually running again

That is useful in cases where:
- a daemon/helper crashes and relaunches under `launchd`
- a request family begins failing around interruption/invalidation boundaries
- proxy setup and listener acceptance are easy to observe but the true service-owned reducer is still missing
- compare pairs keep overclaiming from route or recovery evidence

The practical result is a better stop rule:
- freeze the smallest truth object that actually predicts later behavior
- prefer reply/error class, exported-object re-entry, or one later durable effect over more client-side proxy inventory
