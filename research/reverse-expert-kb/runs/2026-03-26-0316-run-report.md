## New findings
The main practical refinement preserved this run is:
- for Windows exception-owned cases involving generated code or runtime-installed unwind ownership, the branch should not flatten everything into only:
  - landing truth
  - resume truth

It should preserve one extra middle object:
- **range ownership truth**

That means the practical chain is now:
- landing truth -> where exception ownership becomes re-findable
- range ownership truth -> which runtime-installed or callback-owned unwind range actually owns the current PC
- resume truth -> which later resumed target, unwind consequence, or handler-owned edge first predicts behavior

This matters because it fixes two common failure modes at once:
1. stopping too early at `KiUserExceptionDispatcher` / `RtlDispatchException`
2. stopping too early at `RtlInstallFunctionTableCallback` / `RtlAddFunctionTable` / growable-table registration

The KB now preserves a smaller operator stop rule:
- freeze one installed range first
- then freeze one later resume/consequence edge
- then leave broad exception theory

A second useful refinement preserved this run:
- when generated-code unwind ownership is involved, one convenience stack-capture surface under-reporting the frame is weaker evidence than direct unwind-aware signals such as:
  - `RtlLookupFunctionEntry`
  - debugger/ETW-visible unwind success
  - callback-owned range correlation

So the branch now better preserves the rule:
- do not overread one misleading backtrace when runtime-installed unwind ownership is otherwise well supported

## Source-backed synthesis
The useful cross-source synthesis here is:
- Microsoft docs establish the formal mechanics: active function-table lookup, callback-installed ownership, and runtime-added tables for generated code
- Elmo’s SEH/unwind writeup helps explain why dispatcher/unwind reasoning naturally leads into dynamic-function-table ownership when static image metadata is insufficient
- momo5502 reinforces that dispatcher-side landing realism matters, because a wrong landing model can make later unwind ownership look broken when it is actually coherent
- Sebastian Schöner adds a practical workflow caution: dynamic unwind ownership can be real even if one convenience stack-capture path fails to reflect it

The resulting branch memory is sharper:
- not just “dynamic function tables exist”
- but “freeze one installed owning range, then one later behavior-bearing consequence”
