## New findings
The main source-backed takeaway preserved this run:
- BITS-job abuse should not be flattened into one vague persistence story
- the practical proof chain is often:
  - job creation / owner truth
  - resume or activation truth
  - notify-command / callback registration or transferred/error progression truth
  - final callback / launched command / final-availability truth
- callback object registration, callback CLSID persistence, notify command lines, and `Complete()` / final file availability are distinct proof boundaries that deserve to stay separate in the malware branch

What was still missing:

After this run, the malware branch better preserves a practical operator chain for BITS abuse:
- job owner / GUID truth
- activation / resume truth
- notify semantics or transferred/error progression truth
- final callback / launched-command / final-availability truth
- later effect suitable for handoff or reporting

That is practical because it helps analysts avoid:
- overclaiming from job creation alone
- overclaiming from `SetNotifyCmdLine` or callback registration alone
- flattening transferred-state truth into final file truth
- reopening generic packet hunting when the real proof object is durable job ownership or completion semantics
