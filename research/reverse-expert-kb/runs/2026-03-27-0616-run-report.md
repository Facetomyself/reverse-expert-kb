## Practical synthesis added this run
The main KB contribution from this run is a more truthful firmware/protocol stop rule around hardware-facing kicks:
- prepared or published work is not the same thing as notifying the peer
- notifying the peer is not always the same thing as proving the device has actually observed the write yet
- that observation boundary is still weaker than a later completion/status/ISR/worker consequence proving the kick mattered

Useful branch shorthand:
- `published != doorbelled != observed-by-device != consequence`

That shorthand is practical because it tells the analyst what to freeze next instead of broadening into more generic barrier language or broader register cataloging.
