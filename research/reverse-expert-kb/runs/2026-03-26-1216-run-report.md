## Reflections / synthesis
This run materially improves a practical protocol replay question that often causes wasted effort:

Before this run, the KB already preserved:
- method/path/opnum identity
- representative replay-fixture selection
- streaming-shape realism
- Windows RPC binding/context comparability

What still needed stronger branch memory:
- a cross-family compare rule for cases where the body is already plausible but the call-context is still false
- a reminder that not every replay miss is immediately a replay-gate or schema problem

After this run, the branch more honestly preserves a smaller ladder:
- recover one representative method-bearing contract
- externalize it
- freeze one fixture with route/body truth
- freeze one decisive call-context contract when it materially changes comparability
- only then overread later accept/reject behavior as payload or gate evidence

That is practical because it helps analysts avoid:
- reopening broad schema doubt when the real difference is deadline/metadata/binding/context posture
- misfiling comparability failures as later acceptance-gate failures
- treating “same bytes on disk” or “same arguments in notes” as enough proof that the replay object is actually truthful

## Concrete scenario notes or actionable tactics added this run
Actionable compare rule now preserved more clearly:

```text
If two runs keep the same body / opnum / route core but diverge in
metadata, deadline, authority, credentials, binding family, authn posture,
or context-handle lineage, treat the pair as body-identical but
call-context-different before reopening broad body semantics.
```

Practical gRPC checklist retained this run:
1. route identity: `/{package}.{Service}/{Method}`
2. request/response type pairing
3. stream shape
4. metadata/header set that materially changes the call
5. deadline/timeout posture
6. authority/host routing assumptions
7. credential placement if it lives outside the body

Practical Windows RPC checklist retained this run:
1. interface / binding target / endpoint family
2. opnum identity
3. representative argument bundle
4. authn/auth-service posture when relevant
5. context-handle lineage and interface-validity assumptions
