## New findings
The main practical refinement preserved this run is:
- in Windows/native packed cases, a convincing post-stub jump is not automatically the same thing as the first payload-bearing handoff

The branch should preserve five distinct proof objects:
1. **raw PE entry truth**
2. **startup-owned pre-entry truth**
3. **raw post-unpack transfer truth**
4. **startup-normalized payload handoff truth**
5. **first real consumer truth**

That matters because two different practical subcases recur:

### Subcase A: valid stub exit, invalid payload claim
If the analyst really has found the first strong transfer out of visible stub/decrypt/fixup churn, then:
- that may still be only **raw post-unpack transfer truth**
- the next region may remain dominated by TLS callback replay, CRT/runtime startup, constructor/init-table work, import finalization, or loader-side repair/remap
- the analyst should not overclaim payload truth until one import/module/object/consumer anchor remains useful after that startup-owned work quiets down

### Subcase B: pre-entry startup activity flattened into vague anti-debug color
If TLS callback or adjacent startup-managed work is visible before the analyst’s nominal entry-path expectation, then:
- that is not just “anti-debug flavor”
- it is a real **startup-owned pre-entry boundary**
- collapsing it into raw-entry naming or generic anti-debug chatter weakens later payload-boundary selection

## Source-backed synthesis
The useful cross-source synthesis here is:
- Microsoft’s PE format documentation is enough to preserve `.tls` / TLS startup surfaces as first-class image reality rather than folklore
- the retained TLS-callback source reinforces that callback-owned work can execute before the nominal main-entry path that analysts often want to label as “the start”
- from those two facts, the KB can conservatively preserve a practical workflow rule without inventing loader-internal details from weak evidence

The resulting branch memory is sharper:
- not just “OEP can be tricky”
- but “raw entry, startup-owned pre-entry work, raw unpack transfer, payload handoff, and first consumer are often different proof objects; stop at the smallest one that actually predicts later behavior”

Preserved tactics:
- keep the explicit shorthand:
  - `raw_entry != pre_entry_startup != unpack_transfer != payload_handoff != consumed`
