## New findings
The main practical refinement preserved this run is:
- when `SC_ACTION_RUN_COMMAND` is the selected recovery slot, the branch should preserve **launch-context truth** explicitly

The Microsoft service contract makes this a real operator fact rather than stylistic wording:
- `lpCommand` is executed for `SC_ACTION_RUN_COMMAND`
- the process is created via `CreateProcess`
- the process runs under the **same account as the service**

That means the smaller truthful chain is not merely:
- selected slot -> child appeared

It is:
- selected slot -> service-account-shaped launch context -> child/effect

This matters because it prevents three recurring attribution mistakes:
1. treating the run-command child as generic child noise
2. over-attributing the effect to the nominal service image path alone
3. misreading the later child as generic operator/user-context activity when the real owner is SCM-driven recovery under the service account

A second useful synthesis preserved this run:
- `SC_ACTION_RUN_COMMAND` reachability is still weaker than one proved SCM-visible failure path
- non-crash failure handling via `SERVICE_FAILURE_ACTIONS_FLAG` remains part of the reachability story
- queued restart truth and run-command truth must stay separate when both are plausible in the same case

So the branch now better preserves this operator chain:
- failure classification truth
- count/reset-window truth
- selected-slot truth
- launch-context truth
- later effect truth
