## 2. New findings

### Obfuscation-heavy reversing deserves its own topic family
A major structural takeaway is that obfuscation-oriented RE should not be buried under generic malware analysis or generic decompilation.

This run surfaced enough material to justify a dedicated topic family spanning at least:
- obfuscation detection / characterization
- deobfuscation evaluation
- diffing resilience under obfuscation
- function similarity under semantics-preserving transformations
- packer/unpacking-oriented dataset work

That is a better fit for expert RE because obfuscation changes the analyst’s working conditions, not just the surface syntax of recovered code.

### Quarkslab’s obfuscation dataset looks like a strong anchor for realistic obfuscation-resilience research
The strongest concrete source this run was:
- `quarkslab/diffing_obfuscation_dataset`

Important extracted properties from the repository README:
- realistic projects rather than toy examples: `zlib`, `lz4`, `minilua`, `sqlite`, `freetype`
- two obfuscators: **OLLVM** and **Tigress**
- multiple obfuscation families: control-flow flattening, opaque predicates, arithmetic/data obfuscation, virtualization, copy/merge/split, plus combined schemes
- obfuscation levels from 0% to 100%
- multiple optimization levels
- stripped binaries plus JSON ground truth and exported analysis artifacts
- reported scale of **more than 6,000 binaries** and **more than 8M functions**
- practical note: full download is large (~92 GB)

This matters because it is much closer to an analyst-relevant corpus than many earlier obfuscation datasets based on tiny benchmark programs.

### The dataset also reveals a subtle but important benchmark-design problem
One especially useful detail from the Quarkslab README: compiler optimizations can partially erase or alter an applied obfuscation pass.

That is structurally important for the KB. It means obfuscation datasets cannot be judged only by how many transformations were requested during generation. They also need to be judged by whether the resulting binaries still exhibit the transformation in a way that matters to the analyst.

That suggests a reusable evaluation question for the KB:
- is the benchmark labeling **build-time intent**, or **observable post-compilation behavior**?

### Detection and diffing resilience seem to be diverging into separate evaluation objects
The repository:
- `quarkslab/obfuscation_benchmark_code_artifacts`

explicitly points to two paper lines:
- **obfuscation detection / graph-based semantic analysis**
- **binary diffing resilience on obfuscated programs**

That is useful because it sharpens a distinction that expert reversers care about in practice:
- one system helps decide *what kind of obfuscation is present*
- another helps preserve *matching/diffing/triage ability* despite obfuscation

These should probably be treated as related but separate benchmark families.

### REFuSE-Bench adds a useful perspective: similarity search is part of deobfuscation-adjacent workflow
This run also pulled in:
- **REFuSE-Bench** (*Is Function Similarity Over-Engineered? Building a Benchmark*, NeurIPS 2024)

Important details from the abstract:
- focuses on binary function similarity detection
- tries to fix data duplication and labeling quality issues
- includes **real malware** and **Windows** data
- reports that a simple raw-byte baseline can be surprisingly strong

This matters because expert reverse engineering under obfuscation is often not about perfectly restoring source-like code. Often the practical question is:
- can I still identify similar functions,
- cluster variants,
- diff old/new samples,
- or locate reused code quickly enough to steer the next step?

That makes function similarity a first-class analyst aid in obfuscation-heavy scenarios.

### Fine-grained similarity may matter when whole-function matching becomes brittle
Another useful supporting paper:
- **BinSimDB: Benchmark Dataset Construction for Fine-Grained Binary Code Similarity Analysis** (SecureComm 2024)

Key signal from the abstract:
- benchmark includes smaller code snippets such as **basic blocks**
- explicitly tries to bridge differences caused by platform or optimization changes

This suggests a useful hypothesis for future KB work: under stronger obfuscation, expert workflows may benefit from **partial / local similarity cues** rather than expecting reliable whole-function alignment.

### Deobfuscation itself is becoming more explicitly benchmarked
A particularly important lead this run was:
- **DEBRA: A Real-World Benchmark For Evaluating Deobfuscation Methods**

Even with only metadata-level extraction this run, it appears to be framed around:
- real-world open-source programs
- metric-driven function/target selection
- state-of-the-art obfuscators
- exposing failures that toy datasets miss

This is a strong sign that **deobfuscation benchmarking** is maturing into its own explicit subfield rather than remaining an ad hoc appendix inside broader obfuscation papers.

### LLM-era obfuscation research is branching in two directions
Two particularly useful papers here:
- **MetamorphASM / MAD** (*Can LLMs Obfuscate Code?*, AAAI 2025)
- **JsDeObsBench** (*Measuring and Benchmarking LLMs for JavaScript Deobfuscation*, CCS 2025)

MetamorphASM contributes:
- a large assembly-obfuscation dataset (**328,200 obfuscated samples**)
- explicit techniques such as dead code, register substitution, and control-flow change
- evaluation of whether LLMs can generate and analyze obfuscated assembly

JsDeObsBench contributes:
- a benchmark specifically for **JavaScript deobfuscation**
- coverage from variable renaming through stronger structural transforms
- emphasis not just on simplification but also on **syntax accuracy** and **execution reliability**
- inclusion of JS malware cases

Together, they reveal a deeper pattern:
- LLMs are being benchmarked as **deobfuscators**
- but they are also being studied as potential **obfuscation generators**

That creates a new expert-knowledge question: assistance tools may improve analyst productivity, but model-assisted adversaries may also expand the obfuscation search space.

### Robustness under semantics-preserving transformations is not the same as nominal benchmark accuracy
A very useful 2026 signal:
- **Fool Me If You Can: On the Robustness of Binary Code Similarity Detection Models against Semantics-preserving Transformations** (FSE 2026)

Important extracted details:
- introduces **asmFooler**
- evaluates six BCSD models
- uses eight semantics-preserving transformations
- constructs **9,565 variants from 620 baseline samples**
- shows that small, carefully chosen perturbations can strongly disrupt decisions

This reinforces a key synthesis for the KB:
- a benchmark score on fixed data is not enough
- expert-oriented evaluation must distinguish **nominal success** from **robustness under transformation**

### Packed-binary resources should be tracked separately from generic malware datasets
This run also surfaced practical packer-oriented leads such as:
- `packing-box/docker-packing-box`
- `packing-box/awesome-executable-packing`
- `MalDICT`

These are not all “benchmark papers,” but they matter because packed-binary reversing has a stage that precedes decompilation and semantic recovery:
- detect packer
- classify packing behavior
- decide whether/how to unpack
- dump correctly
- only then move to deeper RE

That suggests the KB should eventually distinguish:
- malware/detection datasets
- obfuscation/deobfuscation datasets
- **packer/unpacking corpora**

## 4. Reflections / synthesis
The strongest structural improvement from this run is that the KB now has a clearer path for treating obfuscation-heavy reversing as a first-class theme.

A useful decomposition now looks like this:
- **obfuscation detection** — what protections/transformations are present?
- **deobfuscation quality** — how well can semantics/readability be restored?
- **diffing/similarity resilience** — can analysts still match and triage related code?
- **packer/unpacking stage** — can the analyst even reach a stable analysis artifact?
- **robustness evaluation** — do methods survive semantics-preserving adversarial transforms?

Another useful synthesis: benchmark realism in this area is unusually fragile. It depends on:
- whether source programs are realistic,
- whether obfuscators are diverse,
- whether transformations survive compilation,
- whether labels reflect observable behavior,
- and whether evaluation focuses on analyst-relevant downstream tasks.

Finally, the LLM angle is becoming impossible to ignore. Obfuscation/deobfuscation research is no longer only about classical transformations and symbolic reasoning; it is also about how model-generated or model-assisted transformations change both attack and defense workflows.
