## New findings
The main practical refinement preserved this run is:
- callback-heavy protected-runtime cases should not flatten everything into one vague story of “the callback did it”

The branch should preserve four separate proof objects:
1. **registration truth**
2. **callback firing truth**
3. **emitted telemetry truth**
4. **enforcement-consumer truth**

That matters because two different practical subcases recur:

### Subcase A: immediate rights-bearing consequence
If an `ObRegisterCallbacks` pre-operation path directly rewrites desired/granted access masks, then:
- the downgraded rights-bearing object may already be the first truthful enforcement consumer
- the analyst should not invent a later queue/service story unless stronger evidence requires it

### Subcase B: producer -> handoff -> later policy
If the callback mainly emits compact telemetry and later behavior is owned by another component, then the practical stop rule is:
- freeze one emitted carrier first (queue node, shared record, IOCTL payload, reducer bucket)
- then freeze one later service/game/policy consumer
- do not stop at callback registration or callback-family inventory alone

A second useful refinement preserved this run:
- process-create callback visibility is often setup/timing truth rather than full policy ownership
- in startup-window cases, registration and early callback execution may only establish later telemetry or setup obligations rather than directly explaining the protected behavior

## Source-backed synthesis
The useful cross-source synthesis here is:
- Microsoft docs establish registration and callback execution semantics for object and process-create callbacks, which is enough to freeze the difference between callback family and actual later ownership
- the architectural anti-cheat writeup reinforces a recurring split between kernel collection, usermode service communication, and later policy/reporting logic, making queue/IOCTL/shared-state handoff a practical downstream object rather than speculation
- the BEDaisy case study reinforces that real protected drivers live inside larger runtime architectures, so one visible kernel surface should not be mistaken for the whole behavioral owner

The resulting branch memory is sharper:
- not just “kernel callbacks matter”
- but “registration, firing, emitted telemetry, and enforcement are distinct proof layers; stop at the smallest one that actually predicts behavior”

## Concrete scenario notes or actionable tactics added this run
Preserved tactics:
- keep the explicit shorthand:
  - `registered != fired != emitted != enforced`
- if a handle pre-operation callback already rewrites granted/desired rights and that object predicts the symptom, stop at that rights-bearing object first
- if the callback mainly produces compact telemetry, prefer the first queue/shared-record/IOCTL/service handoff over more callback-family inventory
- do not overread process-create callback visibility as full policy ownership in startup-window cases
- ask what first object survives callback scope and predicts later behavior better than the callback label itself
