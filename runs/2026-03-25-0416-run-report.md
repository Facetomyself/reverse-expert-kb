## New findings
The main practical finding from this run is that the malware branch needed to preserve one narrower operator split for Windows autorun persistence:

1. **Artifact truth is weaker than startup-live truth**
   - a Run / RunOnce / Startup-folder artifact can exist without yet being the startup path that matters now
   - Microsoft Learn also reminds us that `Run` execution timing and ordering are not a hard immediate guarantee

2. **`StartupApproved` can be the smaller truthful boundary**
   - Windows IR material preserves that `Explorer\StartupApproved\Run` and `Explorer\StartupApproved\StartupFolder` can decide whether a visible autorun artifact is actually enabled/live for startup purposes
   - this makes simple Run-key inventory too weak in some cases

3. **Manual artifact mutation and startup-live state do not behave like a trivial mirror**
   - Windows IR experiments suggest manual add/delete behavior in the base Run key does not map one-to-one into `StartupApproved`
   - that matters because it reinforces a practical stop rule: do not collapse install-time write truth into startup-live truth

4. **This family deserved a real malware leaf, not just a parent-page mention**
   - the malware branch already had specific leaves for task/service/WMI/COM continuations
   - the common Run-key/startup family was still effectively under-reduced despite being operationally common

This materially improves the malware branch because it now preserves a practical continuation for cases where the right question is no longer “what persistence family is this?” but “which autorun entry is still live, and which startup-side launcher path proves it matters?”

## Reflections / synthesis
This run intentionally did **not** create another broad malware parent or another taxonomy page.
It added one thin practical continuation because the branch was missing a common startup-side proof object.

The useful synthesis is:

```text
Run / RunOnce / Startup-folder artifact becomes visible
  -> ask whether it is still startup-live now
  -> reduce the case through StartupApproved / current logon startup semantics when relevant
  -> localize one startup-side launcher path
  -> prove one later Shell / Explorer / child-process effect
```

That is practical operator value because it prevents four recurring mistakes:
- treating Run-key presence as equivalent to durable execution proof
- treating `StartupApproved` as registry trivia instead of a smaller truth boundary
- treating install-time registry writes as the interesting object
- letting distraction or stale autorun entries consume the investigation
