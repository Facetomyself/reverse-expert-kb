## New findings
### 1. iOS needed an explicit trust-path continuation between broad gate diagnosis and owner localization
The branch already separated:
- topology
- normalization

- owner recovery

But it still under-modeled a common real case:

### 2. “Generic trust bypass partly works” is a branch-level routing smell
A recurring practical clue is:
- some requests change under generic hooks or bypass snippets
- the target request family does not

That often means the real problem is not just “more pinning.”
It is:
- mixed transport ownership
- wrong runtime surface
- trust-registration context not localized
- or a later post-trust policy consequence being mistaken for trust failure

That deserved a dedicated workflow note.

### 3. iOS branch quality improves when trust-path work is treated as localization, not a bypass recipe
The new note is strongest where it frames the job as:
- pick one target request family
- classify the owning transport/runtime
- separate routing vs trust vs post-trust failure
- localize trust registration
- identify the first decisive validation boundary

That is more reusable than yet another page organized around tool snippets or universal bypass claims.

### 4. Canonical pages must preserve the new iOS middle stage explicitly
Without synchronization, the new leaf would not be enough.
The subtree guide, mobile subtree guide, parent synthesis page, and root index all needed to preserve the same branch shape, or the trust-path stage would decay back into implicit run-report-only knowledge.
