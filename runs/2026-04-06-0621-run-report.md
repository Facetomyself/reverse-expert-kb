## Practical synthesis added
The new workflow note centers on the reusable ladder:

```text
item/helper artifact exists
  != registered with Service Management / BTM
  != approved / allowed / eligible to run
  != actual session-owned or system-owned launch
  != later malware-owned effect
```

Key practical reminders preserved in the new leaf:
- Login Items / BTM-visible startup objects are not the same mechanism family as plain LaunchAgent/LaunchDaemon persistence
- one visible Login Item or Background Item entry is weaker than proving the relevant helper/runtime object actually launched
- `sfltool dumpbtm` is strong for registered/disposition state, weaker for actual runtime launch truth
- approval/allow state is its own boundary and should not be collapsed into registration or launch truth
- bundled helper presence inside an app is weaker than startup-owned execution proof
- manual helper launch is narrowing/debug truth, not natural startup-path proof
