## New findings
The practical refinement added this run is:
- completion-looking bytes in memory still split into two materially different operator cases:
  1. **coherent shared descriptor memory**
     - primary truth boundary is usually publish ordering, freshness semantics, and reclaim
  2. **streaming / non-coherent DMA-backed visibility**
     - completion-looking bytes can still be untrustworthy until an explicit CPU/device ownership-transfer or synchronization boundary is crossed

That means the analyst should not stop at:
- "the bytes are visible"
- or even "the slot contents look right"

The tighter operator chain preserved this run is:

```text
completion-looking bytes exist
  -> classify memory/trust model
  -> if coherent shared memory: prove publish/freshness boundary
  -> if streaming/non-coherent visibility: prove explicit CPU-trust/sync boundary
  -> then prove consumer trust and reclaim/reuse
```

Another useful operator refinement preserved this run:
- "cache visibility" was too small a phrase for what some cases are really doing
- the more truthful wording is often **explicit CPU/device trust transfer**, with cache invalidation or sync as one concrete manifestation of that broader boundary

## Concrete scenario notes or actionable tactics added this run
Preserved practical tactics:
- after localizing one representative completion slot, explicitly classify the slot as:
  - coherent shared descriptor memory
  - or streaming/non-coherent DMA-backed visibility
- if the second case fits better, freeze the first explicit CPU-may-trust-this-now boundary instead of stopping at the visible bytes
- preserve four separate questions in writing:
  1. were the bytes written?
  2. did ownership/freshness/publish state change?
  3. does this memory class require explicit sync before CPU trust?
  4. what reclaim/reuse fact proves actual consumption?
- treat "cache visibility" as too weak when the real issue is broader CPU/device ownership-transfer semantics
