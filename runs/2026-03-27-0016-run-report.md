## Practical operator value added
This run strengthened a live-case rule that is easy to get wrong:
- a suspicious COM registration is not the same thing as a host actually resolving it
- a host resolving it is not the same thing as the path being compatibility-safe and stable enough to matter
- a successful analyst-driven activation is not the same thing as the repeating startup/logon/browser/helper trigger that owns durability in the case
- and none of those are the same thing as one later host-process load / scriptlet fetch / remote script execution / child launch / malware-stage handoff

That is useful in cases where:
- Procmon yields many missing-`HKCU` `InProcServer32` candidates
- a sample writes per-user COM registration and the analyst wants to stop too early
- one `rundll32 -sta {CLSID}` demo succeeds but the ordinary trigger process is still unproved
- a hijack path looks plausible but host stability still depends on matching threading model, proxying exports, or using the right server-path shape
- TypeLib / `script:`-moniker cases risk being flattened into ordinary DLL-replacement folklore

The practical result is a better stop rule:
- freeze the smallest proof object that really predicts later durable behavior
- prefer one natural trigger-process / resolution winner / stable server-path / later consequence chain over more registry inventory or one-off activation demos
