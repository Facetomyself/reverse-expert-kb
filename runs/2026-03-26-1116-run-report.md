## Practical operator value added
This run improves a small but common GUI reversing decision.

Before this refinement, the native GUI branch already helped analysts separate:
- Win32 message pumps from per-window subclass ownership
- Qt direct slot delivery from queued delivery
- Cocoa/XPC/dispatch visibility from later truthful consumers

But an avoidable Qt ambiguity remained:
- if an event filter is visible, is that already the first consumer, or only an early interception boundary?

After this run, the branch more honestly preserves a smaller split:
- event-filter installation truth
- event-filter execution truth
- filter-owned suppression / rewrite / retarget truth
- pass-through truth leading into later object-handler, direct-slot, or queued-slot ownership

That is practical because it helps analysts avoid:
- overclaiming from `installEventFilter(...)` alone
- overclaiming from an `eventFilter(...)` hit that actually returns `false`
- collapsing event interception and queued slot delivery into one vague “Qt async happened” claim
- stopping at framework visibility instead of freezing one consumer-to-effect chain
