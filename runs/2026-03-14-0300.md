# Reverse Expert KB Run Report — 2026-03-14 03:00 Asia/Shanghai

## 1. Scope this run
This run focused on a different seam from the prior benchmark-heavy passes: **expert workflows, analyst interaction needs, and human/LLM collaboration patterns in reverse engineering**.

The aim was to collect genuinely new material that helps answer questions like:
- how expert reverse engineers actually work in practice
- which parts of reverse engineering remain stubbornly human-centered
- what kinds of automation and LLM assistance fit analyst workflows instead of fighting them
- how the KB should organize knowledge about workflow support, not just benchmark results

This was intentionally kept at the research/synthesis level.

## 2. New findings

### Observational/process studies are a critical missing pillar for the KB
A strong result from this run is that the KB should not be organized only around tools, domains, and benchmarks. There is a distinct body of work on **how reverse engineers actually work**.

A particularly important anchor is:
- **An Observational Investigation of Reverse Engineers’ Processes** (USENIX Security 2020)

Even from accessible summaries and established citations, the paper appears to describe reverse engineering as a staged process involving:
- initial overview-building
- subcomponent scanning / triage
- focused deep analysis

That is useful because it gives the KB a process model grounded in observed expert behavior rather than only in tool capabilities or benchmark task taxonomies.

### Protocol RE automation work reinforces that analyst interaction needs matter
Another high-signal paper from this run:
- **An Investigation of Interaction and Information Needs for Protocol Reverse Engineering Automation** (PRE 2025)

This matters because it suggests protocol reverse engineering automation should not be judged only by final inference accuracy. It should also be judged by whether it supports analyst interaction patterns:
- what evidence the analyst needs to inspect
- where automation should remain editable or steerable
- how inferred structure should be presented for verification
- which missing context blocks confident analyst decisions

This is useful beyond protocol RE. It strengthens a general thesis for the KB: **expert-oriented automation must be evaluated partly by interaction fit, not just task success.**

### Human-LLM teaming in RE is becoming more explicit, but often as “assistant” not “replacement”
This run surfaced several strong signals from 2025 work and practitioner materials that converge on a similar pattern: LLMs are being positioned as **sidekicks, copilots, or workflow accelerators**, not clean replacements for expert analysts.

Key examples:
- **ReCopilot: Reverse Engineering Copilot in Binary Analysis** (2025)
- **VirusTotal Code Insight integration for IDA workflows** (2025)
- **Cisco Talos: Using LLMs as a reverse engineering sidekick** (2025)

Common recurring pattern across them:
- analysts select a function or region of interest
- the model/tool generates a summary or proposed interpretation
- previous accepted analyses can be carried forward as context
- the analyst remains responsible for acceptance, correction, and next-step steering

This is a more specific workflow pattern than generic “LLM for binary analysis.” It is closer to:
- analyst-guided triage
- notebook-accumulated context
- iterative hypothesis refinement
- tool-mediated memory reuse

### Notebook / memory-augmented workflows appear especially important
The VirusTotal post was one of the most operationally useful sources this run.

Its noteworthy ideas:
- the analyst can choose a function for analysis
- the system returns a summary and deeper description
- the analyst can **edit/correct** those outputs
- accepted analyses are stored in a **CodeInsight Notebook**
- future queries include the accumulated notebook context
- analysis can improve over time because prior reviewed findings become persistent context

This is important for the KB because it models a concrete assistance pattern that mirrors real analyst cognition:
- working memory is limited
- context accumulates across many local investigations
- corrections matter as much as initial suggestions
- prior accepted interpretations should shape future assistance

This feels like a strong candidate structural motif for expert RE support systems: **persistent, editable, analyst-curated memory**.

### Cost, context-window pressure, and privacy are first-class workflow constraints
The Talos article is useful less for novelty of basic LLM concepts and more for framing practical constraints that shape RE assistance design.

It emphasizes:
- MCP-driven tool use with disassemblers/decompilers
- need for models that handle structured tool-use prompts
- large prompt/context pressure from tool descriptions plus prior outputs plus code
- tradeoffs between local and cloud models in:
  - confidentiality
  - speed
  - monetary cost
  - hardware cost
- risk that insufficient context windows cause degraded or invalid results

This matters because expert workflow support is constrained by operational realities, not just model quality. A theoretically strong assistant may be poor in practice if it:
- bloats context too quickly
- leaks sensitive binaries to the cloud
- costs too much per iterative analysis cycle
- is too slow for exploratory use

### Immersive/visual assistance work is really about cognition, not visual novelty alone
Another useful seam was:
- **Large Language Models as Visualization Agents for Immersive Binary Reverse Engineering** (2025)

What stands out is not just “VR for reverse engineering,” but the framing around:
- embodied and external cognition
- spatial persistence
- visual organization of call graphs and program artifacts
- reducing cognitive complexity
- aligning generated visualizations with analyst tasks and perceptual design principles

Its pilot-study framing also matters:
- human raters judged clarity, task fit, spatial organization, cognitive load, visual encodings, and correctness
- higher guidance in prompts modestly improved some visualization properties
- semantic context (e.g., meaningful function names) appears to affect usefulness

This suggests the KB should track visualization support under a broader category of **cognitive scaffolding for analysts**, not as a novelty subtopic.

### There is a meaningful difference between benchmarked capability and workflow usefulness
Combining this run’s sources with the previous benchmark-heavy runs yields a stronger synthesis:

- benchmark papers ask whether a model/tool can perform a task
- workflow studies ask how analysts actually sequence work and use evidence
- assistant/tooling case studies ask how generated output gets accepted, corrected, stored, and reused

The gap between those is important. A system can benchmark well and still be awkward in practice if it does not fit:
- analyst triage behavior
- evidence review loops
- trust calibration
- memory accumulation
- privacy/cost constraints

That gap looks central to any serious “reverse expert” KB.

## 4. Reflections / synthesis
A major structural takeaway from this run is that the KB needs a new axis beyond domain and benchmark families: **analyst-support pattern**.

The most useful emerging buckets are:
- observational studies of expert reverse engineers
- workflow/sensemaking models
- interaction needs for automation
- human-LLM teaming patterns
- notebook/memory-augmented analysis
- visualization/cognitive scaffolding systems

This axis feels necessary because reverse engineering expertise is not just the possession of facts or tool fluency. It is also:
- staged investigation
- selective attention and triage
- hypothesis management
- confidence calibration
- context carryover across many partial findings

Another strong synthesis: the best current assistance patterns seem to be **analyst-in-the-loop memory systems**, not one-shot answer engines. That suggests future topic pages should pay close attention to how tools preserve, structure, and reuse analyst-validated context.

Finally, protocol RE work and immersive visualization work both reinforce the same deeper point: support systems should be designed around the analyst’s information needs and cognitive bottlenecks, not only around what the automation can infer autonomously.
