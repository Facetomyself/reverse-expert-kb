## New findings
The main practical finding from this run is that the KB needed to preserve a narrower operator rule for flattened protected runtimes:

1. **Dispatch family changes the right proof object**
   - `switch` cases often allow state-to-case / OBB mapping to be the first truthful milestone
   - `indirect` dispatch often makes a **table index / target-family relation** the real proof object
   - `call` dispatch often makes an **arg-struct field / shared-state contract** the real proof object

2. **Dispatcher recognition and successor recovery are separate milestones**
   - Tigress explicitly supports `--FlattenObfuscateNext`, `--FlattenConditionalKinds=compute`, and `--FlattenConditionalKinds=flag`
   - that means a case can have an obvious dispatcher but still legitimately block on successor truth

3. **Dispatcher-local mechanics may still matter even after the target family is known**
   - the KB should preserve a distinction between:
     - `successor truth`: which target family is reached
     - `dispatcher contract truth`: which dispatcher-local writes/lookups/normalization must survive for that edge to stay valid

4. **The right stop rule is often not “fully solve the block”**
   - in indirect/call-dispatch cases, one trustworthy successor family plus one small dispatcher contract note is often enough to leave broad flattening work

This materially improves the KB because it turns a previously implicit idea into an explicit practical continuation rule rather than leaving the branch at generic “table/index normalization matters.”

## Reflections / synthesis
The useful synthesis from this run is:

```text
flattening already recognizable
  -> identify actual dispatch family
  -> choose smallest stable recovery object for that family
  -> recover one trustworthy successor family
  -> write one dispatcher contract if dispatcher-local mechanics still matter
  -> leave broad flattening work for calmer CFG repair / helper cleanup / outer-consumer proof
```

This is a good fit for the reverse KB because it is:
- practical
- case-driven
- conservative
- useful for breakpoint / patch-boundary / hook-placement decisions
- directly relevant to underfed deobfuscation continuations rather than only top-level taxonomy

It also helps prevent two bad habits:
- treating `dispatcher found` as equivalent to `next-state recovered`
- deleting the dispatcher too early once one target family looks plausible
