## New findings (conservative)
Retained practical ladder:

```text
rule artifact exists
  != effective rule after precedence/override
  != matching device event/property set in the run that matters
  != helper/result or selected action path
  != later malware-owned execution/effect
```

Key retained operator consequences:
- a suspicious file in `/etc/udev/rules.d` is weaker than the effective rule set after same-name replacement and lexical order are frozen
- one plausible device family is weaker than one actual `ACTION` / property-set match in the run that matters
- `PROGRAM==`, `RESULT`, and `IMPORT{program}` can be the truthful branch object rather than only the final `RUN+=` string
- `RUN` is restricted to very short-running foreground tasks inside the default `systemd-udevd` sandbox, so direct long-running execution claims should be treated skeptically
- when long-running behavior is intended, `SYSTEMD_WANTS` handoff can be the more truthful durable consumer than `RUN+=` itself
