## New findings
The source-backed practical refinement added this run is:
- malware Scheduled Task persistence analysis should preserve **conditions truth** as its own proof boundary

The narrower chain now preserved is:

```text
definition truth
  -> live-scheduler truth
  -> conditions truth
  -> later launch / callback / malware-stage effect truth
```

What “conditions truth” means here:
- whether task settings make the task actually expected to run under the current machine state
- especially when the decisive settings include:
  - idle-only execution
  - battery start restrictions
  - network-availability requirements
  - delayed catch-up semantics after a missed time boundary

Practical operator consequence:
- a task can be well-formed and scheduler-visible yet still not be expected to fire in the current run
- present-day no-run should not automatically falsify the task chain
- present task definition plus loaded state should not automatically be overread as “therefore it should have run now”

The useful synthesis is:

- loaded-state truth is stronger than artifact-only truth but still weaker than execution expectation under current machine conditions
- settings-level conditions can be the real missing proof boundary in laptop-heavy, compare-run, or field-investigation cases
- therefore the honest reusable object is not just trigger/action/principal/live-scheduler, but trigger/action/principal/live-scheduler/**conditions**/effect

## Concrete scenario notes or actionable tactics added this run
Preserved practical tactics:
- if a task is present and plausibly loaded but did not fire, check settings-level conditions before abandoning the task chain
- keep these separate in case notes:
  1. definition truth
  2. live-scheduler truth
  3. conditions truth
  4. later effect truth
- when comparing runs, include:
  - battery state
  - idle state
  - network availability
  - missed-time catch-up / delayed-start semantics
- do not flatten `StartWhenAvailable` into “it ran on time” or “it definitely should have run immediately”
- do not reopen broad persistence-family inventory if one narrower conditions boundary already explains the no-run discrepancy
