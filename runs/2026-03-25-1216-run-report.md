## Sources used conservatively
Primary retained sources:
- Microsoft Learn — `GetQueuedCompletionStatus`
- Microsoft Learn — `PostQueuedCompletionStatus`
- Microsoft Learn — `CreateThreadpoolIo`
- Raymond Chen — *The mental model for StartThreadpoolIo*

Retained source-backed cues:
- `PostQueuedCompletionStatus` posts packets whose returned values are simply what the caller supplied, and Microsoft explicitly states the system does **not** validate those values and that `lpOverlapped` need not point to a real `OVERLAPPED`
- `GetQueuedCompletionStatus(FALSE)` with non-NULL `lpOverlapped` can still mean a completion packet was dequeued, just for a failed I/O completion rather than a timeout/no-packet case
- `completion key` and `OVERLAPPED*` should be treated as different ownership carriers rather than flattened into one owner claim
- `CreateThreadpoolIo` alone is not callback-delivery proof; `StartThreadpoolIo` is required, and Raymond Chen’s explanation sharpens the operational model: it is conceptually **per I/O operation**, not merely once per bound handle
- under `FILE_SKIP_COMPLETION_PORT_ON_SUCCESS`, immediately successful overlapped operations can skip the callback path and instead require `CancelThreadpoolIo`

## Practical operator value added
This run improved a real analyst stop rule.

Before this refinement, the native completion-driven branch could still allow a slightly too-flat reading:
- a worker thread dequeued something, so it must be the I/O-owned request we care about
- `CreateThreadpoolIo` exists, so the callback path must be live
- the callback did not fire, so the consumer is probably dead or unrelated

After this refinement, the branch more honestly supports:
- **queue/dequeue visibility** -> what was delivered to the worker
- **posted control-packet vs real I/O-owned packet truth** -> what kind of packet it actually was
- **completion-key family truth vs `OVERLAPPED*` owner truth** -> what ownership layer is being identified
- **per-operation callback expectation setup truth** -> whether the thread pool was actually primed for this I/O
- **callback absence under success-skip mode** -> whether the missing callback is a real absence or a notification-mode artifact

That changes real case handling:
- worker-loop visibility is less likely to be overclaimed as concrete request ownership
- control packets posted for shutdown/control flow are less likely to be narrated as real I/O completions
- failed completions are less likely to be discarded as “nothing happened”
- missing `TP_IO` callbacks are less likely to trigger dead-code or wrong-consumer detours when the real problem is expectation setup or immediate-success skip mode

This is practical operator value because it tells the analyst what **not** to conclude too early, which is often the real bottleneck in completion-driven Windows cases.
