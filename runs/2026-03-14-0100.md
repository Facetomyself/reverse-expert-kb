# Reverse Expert KB Run Report — 2026-03-14 01:00 Asia/Shanghai

## 1. Scope this run
Focused on a currently underdeveloped axis in the KB: public benchmarks, datasets, and corpora relevant to reverse engineering.

The goal this run was not to rehash generic tool comparisons, but to identify newer and more evaluation-oriented material that helps answer:
- how RE expertise is being measured
- which public corpora are shaping the field
- how benchmark structure differs across decompilation, binary understanding, firmware, malware, and protocol RE

## 2. New findings

### DecompileBench looks like a foundational anchor for this KB
The strongest concrete new item is **DecompileBench** (ACL Findings 2025), which evaluates decompilers in a far more workflow-relevant way than older synthetic micro-benchmarks.

Notable extracted details:
- 23,400 functions from 130 real-world programs
- runtime-aware validation
- LLM-as-judge for human-centric assessment
- six traditional decompilers + six LLM-powered approaches

Why this matters: it reframes decompiler evaluation around **semantic fidelity plus analyst usefulness**, not just prettiness of pseudocode.

### BinMetric broadens the landscape from decompilers to analyst-like binary tasks
**BinMetric** appears to be a complementary benchmark family rather than a competitor to DecompileBench.

Extracted details:
- 1,000 questions
- 20 real open-source projects
- six binary-analysis task families
- explicit focus on evaluating LLMs on realistic binary-analysis tasks

This is useful because it suggests the KB should distinguish:
- decompiler-output evaluation
- binary-analysis task evaluation
- human workflow support evaluation

### There are likely two distinct “DecompileBench / Decompile-Bench” lines worth separating
Search results strongly suggest the KB should explicitly separate:
- **DecompileBench** = evaluation framework for decompilers in real-world scenarios
- **Decompile-Bench** = million-scale binary-source function-pair dataset for decompilation research

This distinction matters. One is closer to **benchmark/evaluation methodology**; the other is closer to **training or large-scale paired corpus infrastructure**.

### Symbol/type/signature recovery is emerging as a standalone benchmark family
Search results surfaced:
- **R3-Bench** for symbol recovery
- a 2025 benchmark on **binary type inference in decompilers**

This is important because expert reversing quality often depends less on raw pseudocode alone and more on whether names, prototypes, structures, and types are reliable enough to drive subsequent reasoning.

### Firmware RE should be tracked as a corpus-and-context problem, not just an architecture problem
Useful leads surfaced for firmware-specific work:
- sound firmware corpora
- linux firmware corpora
- firmware reverse-engineering workshops/materials
- a 2025 ACSAC paper on recovering peripheral maps and protocols to expedite firmware reverse engineering

Provisional synthesis: firmware RE evaluation differs from ordinary binary analysis because it depends on more than code recovery. It also depends on environment reconstruction, hardware/peripheral understanding, and protocol context.

### Protocol reverse engineering has a fairly different evaluation pipeline
Protocol RE search surfaced recent material on:
- hierarchical protocol reverse engineering
- message field inference
- protocol structure reconstruction
- LLM-assisted protocol analysis (e.g. ChatPRE / ProRE leads)

This suggests protocol RE should probably sit under its own top-level topic cluster instead of being buried as a sub-bullet under “network stuff.”

### Malware datasets are abundant, but many are RE-adjacent rather than RE-native
Leads such as EMBER2024, AU-PEMal-2025, CIC-DGG-2025, and Binary-30K are useful, but many appear optimized for malware detection/classification rather than expert reverse-engineering workflows.

That distinction is worth preserving in the KB:
- some datasets are for classifier benchmarking
- some are for analyst or decompiler benchmarking
- some can be repurposed as corpora but were not designed for RE expertise evaluation

## 4. Reflections / synthesis
The most useful higher-level conclusion from this run is that **reverse-engineering knowledge should not be organized only by tool or platform**. A second organizing dimension is becoming clearer: **evaluation object**.

Candidate evaluation-object taxonomy:
- decompiler output quality
- type/signature/symbol recovery quality
- task-level binary understanding
- firmware corpus realism and environment reconstruction
- protocol message/state inference
- malware corpus utility for RE versus utility for detection

This is more promising than a flat list of tools because it maps better to what experts actually care about: whether a representation is trustworthy enough to support the next investigative decision.

Another useful insight: newer work increasingly treats human usefulness as first-class. That aligns well with the long-term KB goal of modeling expert RE thinking rather than merely cataloging software.
