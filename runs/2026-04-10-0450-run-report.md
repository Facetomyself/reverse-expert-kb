## New findings
Primary durable split reinforced this run:

```text
exported-object method entry
  != local barrier-drained or callback-queue progress truth
  != replied-or-errored exactly-once truth
  != interrupted truth
  != invalidated truth
  != later reconnected/new-request truth
  != same-request durable consequence
```

Concrete practical reinforcements preserved in the KB:
- `remoteObjectProxyWithErrorHandler(...)` plus a reply handler yields one exact-one reply-vs-error classification surface, but that is still weaker than durable consequence truth
- `scheduleSendBarrierBlock(...)` / local send progress is weaker than remote receipt; Apple’s own public comments say waiting for a reply is the better receipt proof
- interruption and invalidation are not the same lifecycle object; interruption may allow later re-establishment, invalidation is a stronger end-of-life boundary for the current connection
- later healthy communication after interruption/invalidation is retry/reconnect truth first, not automatic same-request completion truth
- invalidation can happen before meaningful semantic work even begins, including invalid service-name and listener-rejection shaped cases
