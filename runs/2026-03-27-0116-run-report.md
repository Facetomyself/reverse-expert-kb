## Practical operator value added
This run strengthened a real Apple-platform service/daemon stop rule that is easy to get wrong:
- accepted connection truth is not reply truth
- reply truth is not invalidation truth
- invalidation truth is not reconnection truth
- reconnection truth is not consumer truth
- and none of those are the same thing as one exported-object method or later durable service-owned reducer actually owning the later effect

That is useful in cases where:
- a client-side proxy path looks stable but the service restarts or reconnects under `launchd`
- invalidation fires and the analyst still wants to narrate later activity as if the same proof object continued
- helper/daemon cases produce visible lifecycle recovery before they produce one durable service-owned effect
- compare pairs overread “connection is healthy again” as if that already proved the same semantic path re-ran

The practical stop rule preserved by this run is:
- freeze the smallest proof object that actually predicts later behavior
- in Apple XPC cases, prefer one exported-object method or later durable service-owned effect over generic proxy, lifecycle, or recovery visibility
