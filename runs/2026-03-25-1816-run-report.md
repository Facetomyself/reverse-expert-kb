## New findings
The practical refinement added this run is:
- when `SC_ACTION_RESTART` has already been queued by SCM, later manual restart/stop activity before the delay expires does **not** automatically cancel that queued restart
- therefore one later service reappearance can still belong to an **earlier queued recovery slot**, not necessarily a fresh crash/failure-selection event

This matters because restart-loop cases can otherwise be misread as:
- repeated fresh recovery decisions
- or simple operator/analyst-driven restarts
- when the truthful explanation is still an earlier queued SCM-owned action

A smaller operator chain preserved this run is:

```text
service failure becomes SCM-visible
  -> recovery slot is selected
  -> delayed restart is queued
  -> later analyst/operator-visible restart/stop activity occurs
  -> queued restart still fires after delay
  -> later service reappearance belongs to earlier queued action, not necessarily fresh recovery selection
```

## Concrete scenario notes or actionable tactics added this run
Preserved practical tactics:
- if a service was configured with delayed `SC_ACTION_RESTART`, do not assume a later restart after manual/operator handling is a fresh recovery-selection event
- prove whether the restart had already been queued before the visible intervention
- when restart loops are visible, preserve one exact chain of:
  - SCM-visible failure
  - failure count / reset-window state
  - selected action slot
  - queued-action state
  - later restart or run-command effect
- if that queued-action step is missing, the branch is often not yet ready for broad “service recovery owns persistence” wording
