Conservative retained takeaways:
- epoll separates interest-list registration, ready-list truth, one `epoll_wait(...)` return, and later drain/rearm truth
- `eventfd` readability only proves counter truth, not which loop-side consumer drained it
- `timerfd` readability only proves expiration-count availability, not that the application-level timer handler already ran
- `uv_async_send(...)` is safe cross-thread but libuv coalesces sends, so raw send count is weaker than callback-frequency truth
- practical RE workflow therefore benefits from a narrower Linux-reactor stop rule instead of flattening registration/readiness/wakeup/delivery/consumer truth together

## Practical synthesis added this run
The main KB contribution from this run is a sharper stop rule for Linux reactor-shaped native cases:
- registration is not readiness
- readiness is not one `epoll_wait(...)` return
- one return is not full drain/rearm truth
- wakeup/send visibility is not callback-delivery truth
- callback delivery is still weaker than the later consequence-bearing consumer

This is practical because it changes the analyst’s next move.
Instead of broadening Linux event-loop taxonomy, the workflow can now freeze one narrower edge such as:
- first drain-to-`EAGAIN`
- first `EPOLLONESHOT` rearm
- first loop-thread async callback delivery
- first downstream state write after `eventfd` / `timerfd` drain
