## 2. New findings

### Firmware RE is bottlenecked by hardware-context recovery, not just code recovery
The strongest new finding is structural.

Recent sources suggest that expert firmware reversing often fails or stalls because analysts do not know:
- which peripherals are actually used,
- how registers map to real hardware behavior,
- which protocols sit on top of those access patterns,
- and which parts of the nominal hardware can be ignored safely.

That makes **context recovery** a first-class evaluation object, not just a supporting detail under firmware corpora.

### ProtoReveal looks like a strong anchor for firmware/protocol context recovery
The clearest new source this run was the ACSAC 2025 preview for:
- **Recovering Peripheral Maps and Protocols to Expedite Firmware Reverse Engineering**

High-signal reported details from the preview abstract:
- introduces **access chains** to identify which protocol is implemented for each set of in-use peripheral registers
- prototype: **ProtoReveal**
- evaluation scale:
  - **412 firmware samples**
  - **6 manufacturer websites**
  - **35 microcontrollers**
  - **82 protocols**
  - **ARM and MIPS**
- reported protocol-identification accuracy: **92%**
- reported downstream payoff: reducing automatic rehosting time for fuzzing by **99%** without sacrificing effectiveness by skipping modeling of unused peripherals/protocols

This matters because it directly ties reverse-engineering progress to **environment reconstruction** and **rehosting payoff**, not merely to better disassembly.

### Peripheral-map recovery and protocol identification should be tracked separately
A useful precursor surfaced through:
- `OSUSecLab/AutoMap`
- the linked RAID 2022 paper on revealing hidden memory mapping for peripheral modeling

Even from the accessible README alone, the key point is clear: firmware analysis may first require recovering **hidden memory mappings between peripheral registers** before a higher-level protocol story is even available.

That suggests a sharper decomposition for this KB:
- **peripheral/MMIO recovery**
- **protocol identification over peripheral access patterns**
- **used-vs-unused hardware discrimination**
- **downstream rehosting/emulation acceleration**

### Protocol RE should not be modeled as only field inference
This run also strengthened the protocol side with two complementary sources.

From **BinPRE** (CCS 2024):
- protocol RE without source code still hinges heavily on **field inference**
- but the paper explicitly separates:
  - format extraction
  - semantic inference
  - downstream utility
- reported results:
  - **0.73 perfection** on format extraction
  - semantic inference F1: **0.74** for types, **0.81** for functions
  - **5–29%** higher fuzzing branch coverage than the best prior PRE tool
  - enabled discovery of **one new zero-day vulnerability**

This is especially valuable because it links protocol inference quality to a concrete analyst/security outcome rather than only intrinsic accuracy.

### State-machine inference is a separate protocol-RE stage, especially in mixed environments
The paper:
- **Automatic State Machine Inference for Binary Protocol Reverse Engineering**

adds another useful structural correction.

Its abstract explicitly argues that many PRE methods focus on field identification within a single protocol and neglect **protocol state machine (PSM) analysis**, especially in **mixed protocol environments**.

That makes it clearer that the KB should treat protocol RE as a multi-stage family:
- protocol separation / clustering
- field boundary extraction
- semantic field/function/type inference
- **state-machine inference**
- downstream utility such as fuzzing, traffic understanding, and abnormal-behavior analysis

### False positives matter a lot in protocol-context work
The **BinaryInferno** source was helpful here because it emphasizes a user-facing property that is easy to underweight: **false-positive burden**.

Reported NDSS page metrics for top-level protocols:
- precision **0.69**
- recall **0.73**
- false positive rate **0.04**

Its framing is useful: semantically meaningful descriptions with fewer false positives may be more valuable to users than noisier broad extraction.

That fits expert RE well. In protocol and firmware-context work, a wrong guessed field or wrongly modeled peripheral can waste a surprising amount of analyst time downstream.

### A new distinction is emerging: artifact understanding vs environment reconstruction
The clearest synthesis from this run is that the KB should probably distinguish between two adjacent but different kinds of reverse-engineering success:

- **artifact understanding**
  - what function/message/state does this artifact represent?
  - what are its fields, semantics, and transitions?
- **environment reconstruction**
  - what hardware/protocol context must exist for meaningful analysis to proceed?
  - which peripherals, registers, and protocols are actually in play?
  - what can be safely ignored?

That distinction feels especially important for firmware reversing.

## 4. Reflections / synthesis
The KB structure is improving in a way that feels closer to real expert practice.

Earlier runs established decompilation, obfuscation, and symbol/type recovery as distinct evaluation families. This run suggests that **firmware / protocol context recovery** also deserves explicit first-class treatment.

A particularly useful decomposition now looks like:
- **decompilation evaluation**
- **symbol/type/signature recovery**
- **task-level binary understanding**
- **obfuscation/deobfuscation/diffing resilience**
- **firmware / protocol context recovery**

And inside firmware/protocol context recovery, a second decomposition is emerging:
- peripheral/MMIO/register recovery
- protocol identification
- field/semantic inference
- state-machine recovery
- downstream rehosting/fuzzing payoff

Another useful synthesis: this area makes **downstream utility** hard to ignore. Protocol and firmware recovery research becomes much more compelling when it shows:
- improved fuzzing branch coverage,
- bug discovery,
- faster rehosting,
- or reduced analyst modeling burden.

That is more valuable for an expert KB than collecting isolated accuracy claims.
