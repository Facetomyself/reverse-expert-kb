## New findings
### 1. APC queue visibility is weaker than delivery truth
The source set reinforced a narrow but important workflow claim:
- `QueueUserAPC` / `NtQueueApcThread` visibility is only production truth
- ordinary user APCs still depend on the target thread entering alertable state or calling `NtTestAlert`
- analysts should stop widening queue-site inventory before proving one truthful delivery boundary

### 2. `KiUserApcDispatcher` is usually a reduction boundary, not the final owner
The APC internals sources are especially useful here:
- `KiUserApcDispatcher` proves user-mode delivery is real
- but the practical owner is often one callback body or one immediate downstream consumer later
- this is exactly the kind of stop rule worth preserving canonically so the branch does not stall at dispatch-trampoline naming

### 3. Cross-process APC folklore can mislead first-consumer proof
The Microsoft documentation and APC internals writeups align on a practical caution:
- cross-process queueing is weaker than it looks because rebasing, cross-bitness, and thread alertability all matter
- queueing alone should not be overread as execution proof
- this makes queued-vs-delivered truth a real practical continuation, not just API trivia

### 4. APC-backed async I/O and deliberate alertable-wait work queues deserve their own continuation memory
The sources support a small but durable branch rule:
- some Windows-native async cases are neither broad callback/event-loop problems nor IOCP/thread-pool dequeue problems
- they are specifically APC delivery problems
- the branch now benefits from preserving that narrower path explicitly
