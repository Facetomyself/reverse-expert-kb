## New findings
### 1. The protected/deobfuscation branch’s main weakness was routing, not lack of practical leaves
The branch already had enough practical notes to support a useful subtree.
The missing piece was a compact guide explaining:
- how to enter the branch
- how to classify the current bottleneck
- how to hand off from one practical note to the next

### 2. The branch now reads better as four recurring operator bottleneck families
The strongest compact formulation after this run is:
- **anchor** noisy trace / dispatcher churn
- **handoff** out of packed or staged bootstrap
- **consume** recovered artifacts through the first ordinary consumer
- **tripwire** the first behavior-changing integrity consequence

### 3. Protected-runtime practical work benefits from the same subtree-guide discipline already used in malware, mobile, and protocol branches
Recent branch-repair runs showed that a subtree guide can add more operator value than another isolated leaf note when a branch already has enough good pages.
This run confirms that pattern for protected/deobfuscation work.

### 4. Branch-balance repair can be practical without creating more taxonomy
The new guide did not expand abstract family labels.
Instead it clarified:
- start points
- next-step sequencing
- early-vs-late bottleneck selection
- handoff from protection-shaped problems back into quieter native/runtime workflow notes
