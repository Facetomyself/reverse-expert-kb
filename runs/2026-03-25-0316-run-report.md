## New findings
The main practical finding from this run is that the native loader/provider branch benefited from a narrower Windows-heavy rule that was not yet explicit enough:

1. **Delay-load helper visibility is a resolution boundary, not an ownership boundary**
   - Microsoft delay-load helper material makes the mechanics explicit enough to preserve the workflow lesson: helper activity proves resolution work happened, but not which caller-side feature or later behavior actually matters

2. **Hookable delay-load paths make helper-centric reasoning weaker than it looks**
   - pre-load, pre-`GetProcAddress`, and failure-hook behavior means the helper frame itself is often less stable than the first retained caller-side pointer/table/provider object that survives resolution

3. **Forwarded-export / API-set truth is still weaker than first-consumer truth**
   - resolving the implementation family is useful, but analysts still need the first caller-side retained use or later dispatch edge before they actually have behavioral ownership

4. **Delay-import metadata can overstate real relevance**
   - phantom or condition-gated delayed imports are a useful reminder that import-table presence and even reachable helper code can still overstate what becomes live in the current run

This materially improves the KB because it preserves a sharper practical split between:
- **resolution truth**: which DLL/export/forwarder family was chosen
- **consumer truth**: which caller-side use first makes that edge behaviorally relevant

That is small, concrete, and directly useful for breakpoint planning and stop-rule discipline in Windows-native reversing.

## Reflections / synthesis
This run did not create a new branch. It strengthened an already-valid native continuation so the KB better preserves one practical operator rule:

```text
helper / import-resolution machinery becomes visible
  -> implementation family becomes more plausible
  -> resolved address / thunk patch / forwarder truth becomes visible
  -> prove the first retained caller-side consumer of that resolved edge
  -> only then widen into sibling imports, fallback DLLs, or async delivery
```

That matters because it helps prevent four recurring mistakes:
- treating delay-load helper success as ownership proof
- treating forwarded-export truth as complete behavioral proof
- treating delay-import metadata as if it were automatically live in the current run
- widening into loader taxonomy before one resolved-edge-to-effect chain is grounded

This is the kind of native maintenance the KB needs more of: narrower, practical, and case-driven, not just top-level branch wording.
