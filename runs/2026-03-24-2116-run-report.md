## Why this change matters
Before this run, the native completion/thread-pool branch already knew that:
- submit order is not trustworthy consumer order in IOCP cases
- posted control packets can masquerade as I/O-owned work
- `GetQueuedCompletionStatus(FALSE)` can still reflect a real failed-I/O delivery
- libuv worker-side and loop-thread completion ownership are different boundaries

After this run, the branch more clearly preserves one smaller operator rule:
- IOCP dequeue truth still has two different ownership questions
- `completion key` often reduces queue family or handle family
- `OVERLAPPED*` often recovers the concrete embedded request/session owner
- therefore the analyst should not stop at “this worker got the packet,” but should reduce one explicit identity chain:
  - queue family / handle family
  - concrete owner recovery
  - first branch or callback that changes later behavior
  - one downstream effect

That improves the KB itself because it sharpens a thinner native continuation and makes future queue/dequeue casework less likely to overread delivery visibility as proof of real ownership.

## Concrete scenario notes or actionable tactics added this run
- in IOCP cases, explicitly write down whether the current proof object is:
  - queue-family identity from `completion key`
  - concrete owner recovery from `OVERLAPPED*`
  - or a later behavior-bearing consumer after one of those recoveries
- if one worker loop handles both real I/O and posted control packets, classify packet family before reasoning about ownership
- if dequeue order looks persuasive, re-check whether the real stable proof object is identity recovery rather than chronology
- if libuv worker-side code looks convincing but the visible effect appears only on the loop thread, prefer `after_work_cb` / later async continuation ownership over reopening worker-body analysis first
