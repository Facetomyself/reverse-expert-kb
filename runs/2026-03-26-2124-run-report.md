Useful retained takeaways:
- gRPC official docs clearly keep metadata in an HTTP/2 header/trailer side channel rather than in the request body
- gRPC official docs also make deadline posture behaviorally meaningful via `DEADLINE_EXCEEDED` and cancellation semantics
- Windows RPC official docs make binding handles part of how calls are actually made
- Windows RPC official docs make context handles part of server-maintained state lineage rather than mere copied scalar arguments

Conservative KB-facing synthesis preserved from those sources:
- the same body is not always the same practical call
- the same opnum + arguments is not always the same replay object
- some replay failures should be classified as **body-identical but call-context-different** before reopening broad schema doubt

- gRPC-style replay where metadata, authority, call credentials, or deadline posture lived outside the protobuf body
