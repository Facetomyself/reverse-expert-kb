## Practical operator value added
This run strengthened a real live-case rule that is easy to get wrong:
- seeing the dispatcher is not the same as understanding the direct next state
- seeing a visible selector/index/field is not the same as proving the actual target family
- proving the target family is still not the same as proving that patching out the dispatcher is safe

That matters in cases where:
- Tigress `indirect` dispatch hides the stable relation in `jtab[next]` and in dispatcher-owned mechanics rather than in one obvious constant write
- call-dispatch flattening turns the real proof object into one arg-struct field or callee family while dispatcher-local side effects still matter
- encoded-state flatteners make visible selector domains weaker than one normalized successor relation
- symbolic execution or IL repair recovers a plausible edge, but patching would still delete lookup, normalization, or side-effect obligations

The practical stop rule preserved by this run is:
- before claiming “next state recovered,” ask whether the real proof object is direct constant, target family, or dispatcher contract
- before claiming “patch-ready,” ask what dispatcher-owned semantics still need to survive
- if `target family + dispatcher contract` already gives the next calmer target, stop there instead of forcing premature exact-constant cleanup
