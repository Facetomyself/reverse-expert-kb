## Why this change matters
Before this run, the KB already knew that:
- descriptor/ring publication matters
- mailbox/doorbell publication differs from local staging
- completion and later consequence should be tied together

After this run, the firmware/protocol branch more clearly preserves a smaller operator truth:
- descriptor presence is not descriptor trust
- completion-shaped bytes are not necessarily fresh completion
- publication must be paired with the right ordering / freshness / cache-visibility semantics
- reclaim or slot reuse is often the downstream proof that the earlier handoff was real

That improves the KB itself because future firmware/embedded runs are less likely to drift into:
- ring-anatomy overdocumentation
- generic interrupt narration
- “bytes exist in RAM, therefore solved” false confidence

## Concrete scenario notes or actionable tactics added this run
- when a queue/ring case already has plausible structure, write down whether the current proof object is:
  - descriptor contents only
  - publish edge only
  - trust/freshness rule
  - non-coherent visibility rule
  - reclaim/reuse proof
- if the same descriptor/completion bytes appear in both successful and stalled runs, prefer compare pairs around the trust-enabling publish/sync edge rather than around field layout alone
- if the slot looks populated but software still ignores it, re-check owner/phase/freshness semantics before widening the descriptor map
- if device writes appear real but software still sees stale state, treat cache invalidation / synchronization helpers as part of the main proof object rather than implementation noise
- once the trust contract is frozen, hand off quickly into one next concrete task:
  - rehosting/model realism
  - narrower MMIO/reclaim proof
  - later ISR/deferred consequence proof
