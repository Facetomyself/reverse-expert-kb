## New findings
The source-backed practical refinement added this run is:
- queue-heavy reverse engineering should preserve **notify-vs-trust** as an explicit distinction whenever the target may rely on ordering, freshness, or streaming/non-coherent ownership transfer

The narrower chain now preserved is:

```text
contents prepared
  -> ownership/sync edge makes contents trustworthy
  -> publish / idx / tail / notify announces availability
  -> peer or CPU consumes under freshness rules
  -> reclaim / reuse / wakeup proves durable consequence
```

Practical operator consequence:
- a queue edge can be structurally visible yet still not be the decisive boundary
- populated completion bytes are weaker than fresh completion bytes
- fresh completion bytes are still weaker than CPU-trustworthy completion on non-coherent paths
- completion is still weaker than reclaim/reuse proof

## Reflections / synthesis

- not another broad ring taxonomy summary
- not another top-level wording cleanup
- not another index-only branch-balance gesture
- but one source-backed operator correction that should improve future case handling

The useful synthesis is:
- notification is weaker than trust
- queue progress is weaker than fresh ownership
- fresh ownership is weaker than CPU-trustworthy completion on non-coherent paths
- reclaim/reuse is still the best compact proof that the earlier handoff mattered

So the reusable object is not just:
- descriptor -> idx -> interrupt

It is:
- contents -> ownership/sync -> publish/notify -> fresh consume -> reclaim/consequence

## Concrete scenario notes or actionable tactics added this run
Preserved practical tactics:
- if a descriptor or completion slot is already visible, ask whether the decisive missing edge is actually a sync/ownership boundary rather than another queue field
- keep these separate in case notes:
  1. contents prepared
  2. ownership/sync boundary
  3. publish/notify boundary
  4. freshness/consume boundary
  5. reclaim/reuse/consequence boundary
- when comparing runs, prefer pairs where:
  - the same slot or descriptor exists in both runs
  - but only one run performs the decisive sync/barrier/ownership-transfer helper or freshness transition
- do not flatten `doorbell`, `avail idx`, `WR_IDX`, or interrupt arrival into “therefore the peer/CPU already trusts the contents”
- do not keep broad descriptor taxonomy work going once one narrower trust contract explains the stall
