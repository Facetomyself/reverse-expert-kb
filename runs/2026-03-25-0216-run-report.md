## New findings
The main practical finding from this run is that the protocol branch already had the right leaf, but it still benefited from sharper cross-family reminders around four operator traps:

1. **Completion-queue delivery is weaker than current owner proof**
   - gRPC-style async queues remind us that the tag/event returned by a queue still has to be interpreted against the lifetime of the per-call object or state carrier that the runtime currently trusts

2. **Callback-path correctness is weaker than waiter-liveness proof**
   - RabbitMQ RPC and Direct Reply-To flows reinforce that correct reply-path arrival plus matching outer correlation is still not the same as proving that the current pending waiter is the one being satisfied

3. **Stable slot/index equality is weaker than phase/owner proof**
   - NVMe-style queue/completion handling is a clean analogy for why a repeated visible index can still be stale noise after wrap or reuse if the trusted phase/owner state changed

4. **Replay fixtures often fail because they preserve bytes but not lifecycle realism**
   - once the branch narrows to generation / epoch / slot-reuse realism, preserving request/reply bytes alone is frequently insufficient; the fixture may also need owner-retire timing, wrap assumptions, or reuse boundaries

This materially improves the protocol branch because it keeps the note grounded in concrete operator questions rather than generic “freshness” wording.

```text

  -> visible identifier still appears to match
```

That is useful because it helps analysts avoid four common mistakes:
- treating correlation correctness as sufficient
- treating slot equality as ownership equality
- treating queue arrival as proof of current waiter ownership
