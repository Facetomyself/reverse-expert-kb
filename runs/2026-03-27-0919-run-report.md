## Practical synthesis added this run
The main KB contribution from this run is a narrower, more truthful stop rule for guard-page / fault-owned protected-runtime work:
- seeing guard/trap configuration is not the same as seeing a delivered fault
- seeing one delivered fault is not the same as proving the mechanism is re-armed or sustained
- seeing the mechanism repeat is still not the same as proving the later resumed branch or consequence that actually matters

Useful branch shorthand:
- `guard configured != first fault != re-armed mechanism != resumed consequence`

That shorthand is practical because it tells the analyst exactly what to freeze next instead of broadening into generic anti-debug folklore.
