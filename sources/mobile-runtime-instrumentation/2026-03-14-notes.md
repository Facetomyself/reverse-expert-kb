## High-signal extracted points

### 1. Mobile RE is unusually dependent on runtime instrumentation as a first-class workflow tool
The OWASP MASTG Frida page is useful not because it is novel by itself, but because it states the tooling model clearly:
- Frida supports Android and iOS runtime instrumentation by injecting a JS runtime into the target process.
- It exposes distinct APIs for:
  - native interception (`Interceptor`)
  - high-granularity tracing (`Stalker`)
  - Android runtime inspection (`Java` bridge)
  - iOS runtime inspection (`ObjC` bridge)
- Frida operation modes matter operationally:
  - injected / server-based on rooted or jailbroken targets
  - embedded via frida-gadget on non-rooted / non-jailbroken targets
  - preloaded modes

This suggests mobile reversing should not be described only as “APK/IPA static analysis plus a few hooks.”
A better model is: **static analysis + runtime instrumentation + platform-access strategy**.

### 2. Stalker is not just a convenience feature; it represents a different tracing philosophy
The OWASP page and Frida Stalker docs together clarify a valuable distinction:
- `Interceptor` uses inline hooking/trampolines and is easier but more detectable and higher-overhead in some scenarios.
- `Stalker` uses dynamic recompilation / copied instrumented blocks, leaving original code untouched.
- Stalker supports thread following at function/block/instruction granularity and explicitly discusses transparency/performance tradeoffs.
- The internals are architecture-aware, with explicit AArch64/mobile relevance.

Why this matters for the KB:
- mobile RE should capture **instrumentation mode selection** as a practical expert decision
- there is a real tradeoff between:
  - fast targeted hooks
  - high-granularity tracing
  - stealth / anti-debug resistance
  - performance overhead

### 3. Mobile RE has a three-layer target model that differs from desktop reversing
The accessible material suggests mobile reversing often spans at least three different runtime surfaces:
- managed / framework layer
  - Java/Kotlin on Android
  - Objective-C/Swift on iOS
- native layer
  - JNI, NDK libraries, C/C++ code, system libraries
- OS/platform mediation layer
  - sandbox, code-signing, entitlements, debugger restrictions, IPC, dyld/dlopen behaviors

This is a useful structural distinction because expert mobile reversing often depends on deciding **which layer to instrument first**.

### 4. iOS reversing is constrained as much by access conditions as by program complexity
The Corellium article is vendor-facing but still structurally useful. It highlights that iOS RE is shaped by:
- code signing enforcement
- hardware-backed protections
- jailbreak restrictions
- anti-debugging and runtime tooling detection
- the growing importance of virtualized environments when physical-jailbreak workflows are constrained

That supports a KB framing where iOS expertise includes not just binary understanding, but also **analysis environment control**.

### 5. arm64e / PAC-era iOS creates a new expert subdomain: mitigation-aware reversing
The Epsilon Security write-up on CVE-2025-31201 is especially valuable as a signal source.
Even from partial extraction, it shows:
- modern iOS reversing increasingly touches PAC-aware behavior
- reverse engineering of Apple security updates may involve:
  - dyld / Mach-O diffing
  - interposing behavior
  - IMP and selector handling
  - authenticated pointers / diversifiers
  - subtle page-permission and loader details
- a practical reversing workflow can now be **mitigation-aware**, not just control-flow-aware

This suggests the KB should eventually track a mobile subtopic like:
- arm64e, PAC, dyld interposition, hardened runtime interactions, and mitigation-aware analysis patterns

### 6. Mobile RE evaluation should probably distinguish “hookability” from “understandability”
A useful synthesis from this pass:
For mobile targets, analyst success often depends on whether the target is:
- instrumentable
- observable at the right runtime layer
- stable under hooks/traces
- resistant to anti-instrumentation

That is different from ordinary static understandability.
A target may be hard to decompile cleanly but easy to instrument for answers; conversely, it may be fairly readable statically but operationally painful because hooks are detected or blocked.

### 7. Toolchain churn itself is a knowledge burden in mobile reversing
The OWASP page notes Frida 17 changes, including bridge packaging/runtime changes and API changes.
That matters because expert mobile reversing depends partly on maintaining a live understanding of:
- bridge/runtime packaging changes
- script/API compatibility
- how custom tooling must adapt
- what breaks when tutorials lag behind the current ecosystem

This is an underrated part of expertise: **toolchain drift management**.
