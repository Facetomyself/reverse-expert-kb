# ali-cloud / CHANGELOG

- 2026-04-04: Diagnosed and partially corrected a host-level DNS failure that was breaking both Docker pulls and ad-hoc Hysteria client startup. Root cause on-host was `systemd-resolved` using DHCP-provided Alibaba internal resolvers `100.100.2.136` / `100.100.2.138`, both unreachable from the machine during the incident. Minimal persistent fix applied: created `/etc/netplan/60-openclaw-dns.yaml` so `eth0` keeps DHCP addressing but ignores DHCP DNS and instead uses `223.5.5.5` / `223.6.6.6`. After apply, local stub resolution via `127.0.0.53` recovered and `dig registry-1.docker.io A` succeeded through the stub. Same-day validation also confirmed a temporary Hysteria client on `ali-cloud` could connect to `oracle-gateway` and expose SOCKS5 on `127.0.0.1:18080`, with egress observed as `129.150.61.78` and successful HTTP 200 responses from GitHub and Google through the tunnel. Remaining unresolved issue: some direct HTTPS / Docker registry paths still stall (`hub.zhangxuemin.work` occasional TLS handshake timeout from dockerd; `registry-1.docker.io` direct IPv4 connect timeout), so DNS was fixed but registry-path reliability is not yet fully solved.
- 2026-03-19: Rechecked `camoufox-remote` for replacement planning. Confirmed it is still the manually maintained Apify Camoufox container on public host port `39222`, using `/opt/camoufox-remote/run_camoufox_server_compat.py` via bind mount and exposing plain `ws://<host>:39222/camoufox` without a documented TLS/auth proxy in front. Replacement hardening work was intentionally stopped before cutover; no remote service change was committed. During that work, host tooling was also clarified: this machine has legacy `/usr/bin/docker-compose` available, not the newer `docker compose` plugin.
- 2026-03-18: `ops-assistant` once emitted a misleading P1 summary line for `ali-cloud` (`reachability or core inspection failure`). Review of raw check outputs showed host reachability was healthy (`host_health` and `systemd_inventory` both succeeded) and the likely issue was only a transient `docker_inventory` SSH/banner timeout during that run. Alert classification was corrected in `ops-assistant` so only `host_health` failures map to P1 reachability failures; partial inspector failures now belong to lower-severity handling.
- 2026-03-15: First-pass inspection completed. Confirmed `ali-cloud` (`106.15.239.221`) is an Alibaba Cloud ECS host running Ubuntu 24.04.3. Active runtime includes `1panel.service` on port `80`, an EasyImages container on `10086`, and a `camoufox-remote` container on `39222`. Identified 1Panel state under `/opt/1panel`, EasyImages compose/data under `/opt/1panel/apps/easyimage2/easyimage2/`, and a likely manual camoufox deployment root under `/opt/camoufox-remote`.
- 2026-03-15: Second-pass inspection documented operational details for 1Panel, EasyImages, and camoufox-remote: 1Panel CLI capabilities and state paths, EasyImages compose/data layout and init behavior, and camoufox-remote websocket endpoint plus wrapper scripts.
