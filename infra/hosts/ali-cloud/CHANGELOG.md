# ali-cloud / CHANGELOG

- 2026-04-04: Diagnosed and corrected a host-level DNS failure that was breaking Docker pulls and ad-hoc Hysteria client startup. Root cause on-host was `systemd-resolved` using DHCP-provided Alibaba internal resolvers `100.100.2.136` / `100.100.2.138`, both unreachable from the machine during the incident. Persistent fix applied: created `/etc/netplan/60-openclaw-dns.yaml` so `eth0` keeps DHCP addressing but ignores DHCP DNS and instead uses `223.5.5.5` / `223.6.6.6`. After apply, local stub resolution via `127.0.0.53` recovered and `dig registry-1.docker.io A` succeeded through the stub.
- 2026-04-04: Same-day follow-up established a persistent local Hysteria egress path for this host. Final shape uses Docker Compose under `/opt/hysteria-egress` with a digest-pinned `tobyxdd/hysteria` client container in `network_mode: host`, exposing local-only SOCKS5 on `127.0.0.1:18080` and managed by `hysteria-egress.service`. Docker daemon was then wired to that local SOCKS5 through `/etc/systemd/system/docker.service.d/proxy.conf`. Validation after the cutover confirmed egress IP `129.150.61.78`, successful registry probe to `https://registry-1.docker.io/v2/` through SOCKS5, and successful `docker pull` for both `hello-world` and `tobyxdd/hysteria:latest`.
- 2026-04-04: Same-day gateway build-out added a first public China-side ingress layer on top of that Oracle egress. A new Docker Compose deployment under `/opt/sing-box-gateway` now runs `sing-box-gateway.service`, exposing authenticated SOCKS5 on `0.0.0.0:2080` and authenticated HTTP proxy on `0.0.0.0:2081`. Both forward to the local Hysteria SOCKS5 (`127.0.0.1:18080`). Validation from the OpenClaw host confirmed both ingress modes exit as `129.150.61.78`; Docker Hub registry probe via SOCKS returned expected `401`, and GitHub via HTTP proxy returned `HTTP/2 200`. The sing-box image was then pinned to digest `ghcr.io/sagernet/sing-box@sha256:8772c662c8e349d3afb0c233ccc3864d7df69ce840d5aa25db4c248d5bcb44f7`, and obsolete Compose `version` keys were removed from both local gateway stacks to reduce future drift/noise. This host is therefore no longer just a host-local foreign egress fix; it now functions as an explicit-proxy transit gateway that domestic machines can use to reach Oracle-side egress.
- 2026-03-19: Rechecked `camoufox-remote` for replacement planning. Confirmed it is still the manually maintained Apify Camoufox container on public host port `39222`, using `/opt/camoufox-remote/run_camoufox_server_compat.py` via bind mount and exposing plain `ws://<host>:39222/camoufox` without a documented TLS/auth proxy in front. Replacement hardening work was intentionally stopped before cutover; no remote service change was committed. During that work, host tooling was also clarified: this machine has legacy `/usr/bin/docker-compose` available, not the newer `docker compose` plugin.
- 2026-03-18: `ops-assistant` once emitted a misleading P1 summary line for `ali-cloud` (`reachability or core inspection failure`). Review of raw check outputs showed host reachability was healthy (`host_health` and `systemd_inventory` both succeeded) and the likely issue was only a transient `docker_inventory` SSH/banner timeout during that run. Alert classification was corrected in `ops-assistant` so only `host_health` failures map to P1 reachability failures; partial inspector failures now belong to lower-severity handling.

Active runtime includes `1panel.service` on port `80`, an EasyImages container on `10086`, and a `camoufox-remote` container on `39222`. Identified 1Panel state under `/opt/1panel`, EasyImages compose/data under `/opt/1panel/apps/easyimage2/easyimage2/`, and a likely manual camoufox deployment root under `/opt/camoufox-remote`.
