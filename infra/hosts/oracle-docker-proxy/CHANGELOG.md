# oracle-docker-proxy / CHANGELOG

- 2026-03-15: Created placeholder host documentation from Cloudflare DNS export only. No SSH verification yet. Initial mapping suggests `129.150.61.78` is a shared `docker_proxy` host serving multiple `*.zhangxuemin.work` names including registry-related endpoints and a UI endpoint.
- 2026-03-15: External probe pass confirmed this host group is fronted by Caddy: `hub`, `ghcr`, `gcr`, and `quay` return `HTTP 200` with `server: Caddy`; `ui.zhangxuemin.work` currently returns `HTTP 502` from Caddy.
- 2026-03-15: Confirmed actual SSH entry via local alias `oracle-docker_proxy` with key `~/.ssh/oracle-docker_proxy`. On-host inspection verified `24-7-10-2055` (Ubuntu 20.04.6, x86_64) runs Caddy on `80/443`, Caddy admin on `127.0.0.1:2019`, and a set of long-lived Docker registry proxy containers mapped through `/etc/caddy/Caddyfile`. Also confirmed the `ui` backend issue is not just edge-facing: local access to `127.0.0.1:50000` resets the connection.
- 2026-03-15: Inspected `/root/harbor/docker-compose.yml`, `registry-ui`, and the registry proxy containers. Found that `registry-ui` targets `reg-docker-hub:5000` and is failing on some manifest fetches with `Faraday::ServerError`, which likely explains the public `ui.zhangxuemin.work` failures. Also documented the registry proxy stack's shared data path under `/data/registry-proxy/registry/data`, per-registry config files under `/data/registry-proxy/registry-*.yml`, and the explicit upstream mapping for each registry mirror domain.
- 2026-03-15: Confirmed Harbor deployment files and `/data` footprint exist on-host, but there are currently no active Harbor containers. The present live service surface is the custom Caddy + `dqzboy/*` registry stack.
- 2026-03-15: Refined `registry-ui` diagnosis. Current evidence says `reg-docker-hub` itself is healthy and can serve OCI index manifests, while `registry-ui` throws `Faraday::ServerError` on some tag/digest follow-up fetches. This makes the `ui.zhangxuemin.work` issue look more like a UI compatibility problem with certain OCI index / manifest-list cases than a Caddy or registry infrastructure outage. Also noted that one earlier `docker exec` failure was a stopped-state race and should not be overinterpreted as a persistent crash loop.
- 2026-03-16: Performed a read-only host health check over SSH. Current snapshot: uptime ~221 days, root disk 38% used, load effectively idle, and the host remains reachable/stable. Main caution is capacity, not outage: it has only ~952 MiB RAM, with low free memory and some swap in use, so it should still be treated as a small-footprint host and not a good place to pile on more services.
- 2026-03-16: Follow-up probe found meaningful front-door/backend drift versus the broader 2026-03-15 picture. Caddy still routes `ui`, `hub`, `ghcr`, `gcr`, `k8sgcr`, `k8s`, `quay`, `mcr`, `elastic`, `nvcr`, and `hubcmd` to local ports, but only `51000` (`hub`), `52000` (`ghcr`), `55000` (`k8s`), and `57000` (`mcr`) answered locally during this pass. The mapped backends for `ui` (`50000`), `hubcmd` (`30080`), `gcr` (`53000`), `k8sgcr` (`54000`), `quay` (`56000`), `elastic` (`58000`), and `nvcr` (`59000`) currently refuse connections, which matches today's external `HTTP 502` results on those routes.
